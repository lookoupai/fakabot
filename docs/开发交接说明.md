```text

```

不能使用：

```text

```

1. 读取 `webhook_secret`。

3. Redis 未命中时查 PostgreSQL。
4. 解密 Token。
5. 获取 Bot 实例。
6. 将 Update 分发给 Dispatcher。
7. Middleware 注入租户上下文。

## 10. 权限约定

平台角色：

- platform admin
- normal user

租户角色：

- owner
- admin

只能 owner 做：

- 修改 Bot Token。
- 删除或停用 Bot。
- 修改提现地址。
- 添加或移除管理员。
- 修改管理员权限。

admin 角色通过 `tenant_role_permissions` 控制细粒度权限：

- `settings`
- `products`
- `orders`
- `payments`
- `finance`
- `supply`
- `api_keys`
- `subscription`

未显式写入权限记录时，admin 默认保持允许，避免升级后破坏现有租户管理能力。

权限必须在 Service 或 Middleware 层校验，不能只靠按钮隐藏。

Tenant Admin API Key 的订阅入口使用独立 scope：`subscriptions:read` 可读本租户订阅状态和订阅账单摘要，`subscriptions:write` 可创建本租户续费订单并尝试生成平台托管支付链接；财务入口使用独立 scope：`finance:read` 可读余额、余额重算审计、提现列表和单笔提现详情，`finance:write` 可发起提现申请；供货入口使用独立 scope：`supply:read` 可浏览供货市场、查看本租户代理申请/代理商品和供应商侧供货/待审申请，`supply:write` 可提交代理申请、创建/上架本租户代理商品、开放供货商品、切换供货审批、审核已有代理申请并设置已有代理关系的独立代理规则；`tenant_admin:*` 仍覆盖所有 Tenant Admin scope。`subscriptions:write` 不代表真实支付联调已完成，`finance:write` 不代表自动打款权限，也不应绕过平台人工审核，`supply:write` 不代表平台审核、平台供货下架恢复权限或真实分账联调已完成。

Platform Admin HTTP API 不复用 Tenant Admin API Key。当前只支持配置型 `PLATFORM_ADMIN_API_KEY_HASHES`，请求使用 `Authorization: Bearer ...` 或 `X-Platform-API-Key`，服务端只比较 SHA-256 hash；未配置 hash 时平台接口默认关闭。`platform_risk:read`、`platform_risk:write`、`platform_finance:read`、`platform_finance:write`、`platform_subscriptions:read`、`platform_subscriptions:write`、`platform_supply:read` 和 `platform_supply:write` 是平台侧语义，不应加入 `TenantApiKey` 的 `SUPPORTED_API_KEY_SCOPES`，也不得加入 `TenantApiKey`。

## 12. 支付实现顺序

第一版优先：

1. epusdt GMPay 平台收款。
2. `epay_compatible` 易支付兼容通道，作为租户自营订单的第二优先可用支付能力。
3. 支付回调验签。
4. 回调幂等。
5. 自营商品发货。
6. 订阅续费。

后置：

- TRC20 直付真实扫链、生产可用直付订单和真实对账迁移；当前已完成 `usdt_trc20_direct` 离线配置骨架、显式自营订单本地离线付款说明页、纯函数解析/匹配核心、离线链上交易落库和本地 pending payment 匹配服务。
- TOKEN188 真实网关联调、真实回调验收和对账查询；当前只完成离线建链、回调和 FakaBot 自定义查单 fixture 归一化合同。
- 柠檬支付和其他支付系统真实网关联调、真实回调验收和对账查询；当前只保留离线建链、回调或 fixture 归一化合同。易支付兼容能力近期只要求按 `epay_compatible` 完成主流程验收，不扩展到更多支付系统。

Tenant Admin 侧已开放 epusdt 兼容配置 API，以及 `epusdt_gmpay`、`token188`、`epay_compatible`、`lemzf`、`usdt_trc20_direct` 通用租户支付配置 API，仅用于保存、读取和停用配置；它不会做保存时连通性测试，也不代表真实支付跳转、回调或对账已完成 staging 联调。`usdt_trc20_direct` 当前额外限制为离线配置、能力摘要、显式自营订单本地离线付款说明页和纯函数解析/匹配核心，不进入自营订单自动 provider 优先级。自营订单默认优先使用租户显式启用的可建链 provider，没有租户级 provider 时回落现有平台 epusdt；代理订单和订阅续费仍强制平台 epusdt，避免资金边界被租户自收款配置覆盖。

原因：

- epusdt 是新需求明确点。
- 平台托管收款和订阅都需要一个稳定优先通道。
- 易支付是当前确认保留的第二通道，其他支付系统先不占用近期开发容量。

## 14. 账本实现顺序

账本不应在第一阶段实现。

顺序：

1. 自营订单先不走账本。
2. 订阅订单走平台收入，但可先简单记录 invoice。
3. 供货代理开始前实现账本。
4. 账本上线前必须有单元测试覆盖分录平衡。

账本必须保证：

- 分录不可变。
- 幂等 key 唯一。
- 冻结期默认 7 天。
- 提现人工审核。
- 退款按实际获得金额扣回。

Tenant Admin 财务/提现 HTTP API 只覆盖余额查询、余额重算审计、提现申请、提现列表和单笔提现详情。余额重算审计复用 `LedgerService.audit_account_balance()`，只对比账户快照与流水重算值，不自动修正余额；提现申请复用账本服务冻结可用余额并创建待审核记录，不执行链上、银行卡或第三方出款；完成和拒绝仍属于平台审核动作。

## 16. 常见跑偏点

避免这些实现：

- 在全局变量里保存当前租户。
- 用 Bot Token 当租户 ID。
- 在 URL 或日志输出 Bot Token。
- 继续把所有逻辑写进一个大文件。
- 用 SQLite 做目标多租户数据库。
- 用 JSON 文件保存订单和用户。
- 支付成功后不加事务直接发货。
- 账本直接覆盖余额，没有流水。
- 文件下载使用永久公开链接。
- 在浏览器端保存 Tenant Admin API Key、Bot Token、支付密钥或外部源凭据。
- 代理商选品上架时不要求选择目标克隆 Bot。
