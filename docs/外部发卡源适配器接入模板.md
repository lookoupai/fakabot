# 外部发卡源适配器接入模板

本文档约束真实第三方外部发卡源 provider 的第一版接入方式。目标是让具体适配器复用平台已有安全边界，不把凭据、卡密、上游错误原文或内部标识带入公开响应、审计和日志。

该模板属于当前阶段的受控 `external_source` 适配器方案，不是 WordPress 式通用插件运行时规范。真实外部货源接入必须先按 provider protocol、manifest 摘要、租户外部源连接、签名 handle、离线合同和脱敏测试推进；当前不支持适配器在线安装/升级、动态前端面板、动态 Bot handler、任意后台任务或插件自带数据库迁移。

当前内置的 `mcy_shop` 仅是离线 fixture provider skeleton：它使用平台定义的 `mcy-shop-fixture/*` 路径和 `mcy_shop_offline_fixture_v1` 合同头验证 adapter 边界，凭据 `base_url` 只允许 loopback、`localhost`/`.localhost` 或 `.test/.example/.invalid` 保留测试主机，不代表真实 mcy-shop/麦卡云接口已完成字段映射、签名、认证、下单、发货或幂等联调。真实接入时必须按本文档重新确认上游规格和测试，不得直接把 fixture 合同当作生产 API。

## 1. Provider 标识

- `provider` 名称只能使用小写字母、数字、下划线和短横线。
- `source_key` 用于区分同一 provider 下的不同上游账号或渠道，也必须走相同标识符规范。
- 不要把上游账号、域名、API Key 前缀或租户信息拼进 `provider` 名称。
- provider 摘要可以暴露 `integration_kind`、`contract_name`、`production_ready` 和 `staging_verified`，但只能是安全标识符和布尔值。真实适配器未完成联调前必须保持 `production_ready=False`、`staging_verified=False`；这些字段不得包含域名、账号、API Key 前缀、连接 ID 或 `source_key`。

## 2. 凭据边界

- 真实凭据只能通过 `ExternalSourceConnectionService.load_runtime_credentials()` 获取。
- provider 必须优先实现 `*_with_context` 方法，并从 authenticated context 中读取 runtime credentials。
- 不允许在 provider 构造函数、全局变量、`raw_payload`、异常文本或 DTO 中保存明文凭据。
- `repr(context)`、`repr(runtime_auth)`、`repr(ExternalHttpRequest)` 和 `repr(ExternalHttpResponse)` 必须保持脱敏。

## 3. HTTP 请求

- HTTP provider 应使用 `ExternalHttpClient(ExternalHttpxTransport(...))`。
- 测试环境必须使用 fake transport 或 `httpx.MockTransport`，不允许单元测试连接真实第三方。
- 不允许绕过 `ExternalHttpRequest` 直接在 provider 中散落 `httpx` 请求。
- 真实 provider 不应手拼 path/query，应使用 `build_external_http_url()` 构造上游 URL；base URL 不带 query，path segment 禁止空值、控制字符、`/`、反斜杠、`.` 和 `..`，query key 去空白后不可重复。
- URL 不能包含用户名、密码或 fragment；敏感值不要放在 query，若上游强制要求 query token，也必须确认错误和日志只出现脱敏后的 URL。
- 默认不跟随重定向；上游 3xx、4xx、5xx 均由 client 转为泛化 `ExternalHttpError`，异常字符串不回显上游 body/header/query，但会提供 `status_code`、`category`、`retryable` 供 provider 做安全分流。

## 4. 返回对象

- 目录同步返回 `ExternalProductPage` 和 `ExternalProduct`。
- 下单和查单返回 `ExternalOrder`。
- 取发货返回 `ExternalDelivery`。
- `raw_payload` 只能保存非敏感摘要，字段名递归不得包含 token、secret、password、apikey、authorization、cookie、credential、session、plain_key、card_secret、storage_key。
- 发货内容只能进入 `ExternalDelivery.items/message`，由 `ExternalDeliveryImportService` 加密导入本地一次性库存；不要写入 `raw_payload`。

## 5. 错误处理

- 上游异常统一转为 `ExternalSourceError` 或 `ExternalHttpError`。
- Tenant Admin HTTP 层对 provider 失败只返回泛化 502，不回显上游响应体、签名串、请求头、卡密或凭据字段名。
- Admin Web 外部源目录同步 BFF 对 provider 失败同样只返回泛化 502；成功响应也只能返回 provider/source、同步计数、next cursor 和本地商品安全摘要，不返回 `connection_id`、外部商品 ID、raw payload、凭据字段、发货内容或 storage key。
- 参数错误应在 provider 调用前由服务层校验并返回调用方错误；不要把租户参数错误伪装成上游 502。
- provider 可以使用 `ExternalHttpError.status_code/category/retryable` 区分凭据错误、外部商品不存在、限流和上游故障，但不能解析或拼接异常字符串来做业务分支。

## 6. 幂等和重试

- 真实 provider 接入前必须确认同一 `out_trade_no` 重试是否返回同一外部订单。
- provider 默认不允许后台自动履约；只有显式声明类属性 `auto_fulfillment_idempotent = True`，并同时实现 context 型下单、查单和取发货能力，registry 摘要 `auto_fulfillment_idempotent_available=True` 后，后台 `external_fulfillment` worker 才允许自动创建外部订单。
- `auto_fulfillment_idempotent_available=True` 只表示 provider 离线合同声明同一 `out_trade_no` 重试幂等，并具备 context 型订单/发货能力；不代表真实第三方已联调或可生产使用。
- 未注册 provider、未 opt-in 或 capability 不满足时，自动履约会在解密运行时凭据和调用 provider 前失败并写泛化审计，`failure_stage=provider_capability`、`failure_category=auto_fulfillment_not_enabled`。
- 如果上游不支持幂等下单，不应启用后台自动履约；需要先引入外部履约状态持久化表，该结构变更必须单独确认。
- 自动履约必须要求本租户 active 外部源连接；连接缺失、disabled、跨租户或 deleted 时直接失败并写泛化审计，不调用 provider。
- 自动履约失败当前通过 `audit_logs` 追踪，不等同于完整外部订单状态机；审计只写安全上下文，如 `stage/category/retryable/status_code/external_order_id/failure_fingerprint`，不写上游 body/header/query、凭据、卡密或原始 payload。
- Tenant Admin 可以通过只读观测入口展示 `external_fulfillment.failed` 审计摘要，但必须做字段白名单：允许订单号、provider/source、外部商品 ID、连接 ID、外部订单 ID、失败阶段、分类、是否可重试、状态码和失败指纹；不得返回完整 `metadata_json`、凭据、卡密、`raw_payload`、上游 body/header/query 或原始异常文本。
- 同一订单同一 `failure_fingerprint` 重复出现时，可以跳过新增 `external_fulfillment.failed` 审计以避免审计洪水；批处理 `failed_count` 仍照常计数。该能力只是审计去噪，不是状态表。
- 审计 `target_id` 使用本地 `order.id`，完整 `out_trade_no` 放入 metadata，避免订单号长度超过审计字段；创建外部订单成功但发货未就绪时，`external_order_id` 只作为人工追踪线索，不代表已建立状态机。

## 7. 最低测试

新增 provider 前至少补齐：

- provider 自身使用 fake transport 或 `httpx.MockTransport` 的目录、下单、查单、取发货合同测试。
- 凭据只进入上游请求，不进入 DTO、`raw_payload`、错误文本或 `repr` 的断言。
- 非 2xx、上游异常、非 JSON 响应、缺字段响应的失败路径。
- 401/403、404、429、5xx 等状态码的 `category` 和 `retryable` 分流断言。
- 与 `ExternalCatalogSyncService`、`ExternalOrderOperationService`、`ExternalDeliveryImportService` 的服务级接线测试。
- Tenant Admin 手动外部发货导入 HTTP 测试应断言 `imported` / `dry_run` 字段，确认 dry-run、真实导入和已有发货记录复用能被响应区分；响应不得回显卡密、凭据、外部 `items/message` 或 `raw_payload`。
- `scripts/staging_readiness.py` 仍必须输出 `fail=0`。
