# 多租户发卡平台数据库设计

## 1. 设计目标

本文档细化 `docs/多租户发卡平台完整方案.md` 中的数据库草案，用于后续编写 SQLAlchemy Model 和 Alembic 迁移。

数据库目标：

- PostgreSQL 作为唯一持久化业务数据源。
- Redis 只做缓存、限流、短锁和 FSM。
- 所有租户业务数据必须能按 `tenant_id` 隔离。
- 支付、发货、账本必须具备幂等和可审计能力。
- 敏感字段只存密文或哈希，不存明文 Token、支付密钥、卡密内容。

## 2. 通用约定

### 2.1 主键和时间

- 主键统一使用 `BIGSERIAL` 或 UUID。第一版推荐 `BIGSERIAL`，公开接口另设 `public_id`。
- 时间字段统一使用 `TIMESTAMPTZ`。
- 所有核心表保留：

```text
created_at TIMESTAMPTZ NOT NULL DEFAULT now()
updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
```

### 2.2 金额

- 金额字段使用 `NUMERIC(20, 8)`。
- USDT 可保存 6 位以上小数，避免链上金额匹配损失精度。
- 展示货币和结算货币分离。
- 历史订单必须保存汇率快照。

### 2.3 状态字段

状态字段第一版可使用 `VARCHAR(32)` + CHECK 约束，后续再评估 PostgreSQL Enum。

原因：

- 迁移简单。
- 状态扩展成本低。
- 更适合快速迭代。

### 2.4 租户隔离

除平台级表外，业务表必须包含 `tenant_id`。

跨租户关系必须显式命名：

- `supplier_tenant_id`
- `reseller_tenant_id`
- `source_tenant_id`

Repository 默认按当前上下文 `tenant_id` 过滤。

当前实现兼容关系：

- `tenants.self_sale_enabled`、`tenants.supplier_enabled`、`tenants.reseller_enabled` 是店铺功能开关的主字段。
- `tenant_settings.feature_flags` 仍保留 `self_sale`、`supplier`、`reseller` 三个运行时兼容 key；运行时读取顺序为默认值 -> `tenants` 三列 -> `tenant_settings.feature_flags` 覆盖。
- Admin Web 店铺设置写入功能开关时会同步更新 `tenants.*_enabled` 与 `tenant_settings.feature_flags`，不允许浏览器直接提交原始 `feature_flags` 或 `clone_enabled`。

### 5.5 trc20_direct_transfers

用途：记录离线录入或 fixture 回放得到的 TRC20-USDT 链上 transfer，并提供本地待支付订单匹配依据。该表只服务离线落库和本地匹配合同，不负责扫链、不发起外部网络请求、不读取 `.env`，也不代表生产直付或真实链上对账完成。

匹配状态：

```text
duplicate_tx
not_confirmed
ambiguous
matched
```

字段：

```text
id BIGSERIAL PRIMARY KEY
tenant_id BIGINT NOT NULL REFERENCES tenants(id)
payment_id BIGINT REFERENCES payments(id)
order_id BIGINT REFERENCES orders(id)
out_trade_no VARCHAR(96)
tx_hash VARCHAR(128) NOT NULL
from_address VARCHAR(128) NOT NULL
to_address VARCHAR(128) NOT NULL
amount NUMERIC(20, 8) NOT NULL
block_number BIGINT
block_timestamp TIMESTAMPTZ
confirmations INTEGER NOT NULL DEFAULT 0
match_status VARCHAR(32) NOT NULL
matched_at TIMESTAMPTZ
created_at TIMESTAMPTZ NOT NULL DEFAULT now()
updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
```

索引：

```text
UNIQUE (tx_hash)
INDEX (tenant_id, match_status, created_at)
INDEX (tenant_id, out_trade_no)
INDEX (payment_id)
INDEX (order_id)
```

说明：

- `Trc20DirectTransfer` / `trc20_direct_transfers` 的第一版目标是离线链上交易落库与本地匹配服务，由 `Trc20DirectReconcileService.record_transfer()` 和 `Trc20DirectReconcileService.match_pending_payment()` 驱动。
- `duplicate_tx` 表示重复交易哈希，不应再次匹配订单。
- `not_confirmed` 表示确认数不足，不应更新订单或支付状态。
- `ambiguous` 表示候选订单不唯一，不应猜测匹配。
- `matched` 表示本地 payment/order 已被明确匹配；这仍不等同于真实扫链、真实 provider 对账或生产可用直付。
- Tenant Admin 转账只读观测合同目标为 `GET /api/v1/tenant/payments/trc20-direct/transfers`，要求 `require_scope("payments:read")`，通过 `Trc20DirectTransferObservationService` 或等价服务按当前租户读取已落库 transfer。OpenAPI 响应模型为 `TenantTrc20DirectTransferListResponse` / `TenantTrc20DirectTransferItem`，只能暴露交易哈希、区块/确认数、合约地址、金额、匹配状态、订单号、时间和 `from_address_masked` / `to_address_masked`；不得暴露 `id`、`tenant_id`、`payment_id`、`order_id`、`raw_payload`、`payload_json` 或 `metadata_json`。该观测入口只读取本地记录，不扫链、不外联、不读取 `.env`，也不代表生产直付或真实链上对账完成。
- 订单排障摘要接入 TRC20 直付 transfer 时只作为 `orders:read 安全聚合`，由 `OrderDiagnosticsService` 查询 `Trc20DirectTransfer` 后在 `trc20_direct` 摘要中返回 `expected`、`transfer_count`、`latest_match_status`、`latest_confirmations`、`latest_matched_at` 和 `latest_amount`。该聚合不得返回 `tx_hash`、`from_address`、`to_address`、`id`、`tenant_id`、`payment_id`、`order_id`、`raw_payload`、`payload_json` 或 `metadata_json`；完整转账摘要仍走 `payments:read` 的 TRC20 转账观测接口。该合同仍不扫链、不外联、不读取 `.env`，也不代表生产直付或真实链上对账完成。

## 10. 关键事务

### 10.1 支付成功事务

1. 根据 `out_trade_no` 查询订单，`FOR UPDATE`。
2. 如果订单已经是 `paid/delivered/completed/refunded`，直接返回幂等成功。
3. 写入 `payment_callbacks`，若 payload 已处理则返回。
4. 更新 `payments.status=paid`。
5. 更新 `orders.status=paid` 和 `paid_at`。
6. 如果是平台托管订单，创建账本 pending 分录。
7. 触发发货服务。
8. 提交事务。

### 10.2 卡密发货事务

1. 查询订单，`FOR UPDATE`。
2. 查询是否已有成功 `delivery_records`。
3. 若无，锁定库存或读取已锁库存。
4. 更新库存为 `used`。
5. 写入 `delivery_records`。
6. 更新订单为 `delivered/completed`。
7. 提交事务。

### 10.3 结算释放事务

1. 查询到期的 `ledger_entries.status=pending`。
2. `FOR UPDATE SKIP LOCKED` 分批处理。
3. 更新 entry 为 `available`。
4. 增加账户 `available_balance`，减少 `pending_balance`。
5. 提交事务。

### 10.4 提现申请事务

1. 锁定 `ledger_accounts`。
2. 校验 `available_balance >= amount`。
3. 创建 `withdrawal_requests`。
4. 扣减 `available_balance`，增加 `frozen_balance`。
5. 创建提现冻结账本分录。
6. 提交事务。
7. 事务后通知平台管理员。

### 10.5 退款事务

1. 锁定订单。
2. 校验订单可退款。
3. 创建 `refunds`。
4. 按各方实际获得金额创建反向账本分录。
5. 更新订单状态。
6. 提交事务。
