# 业务插件架构方案

## 1. 目标

本方案定义 Fakabot 后续业务插件能力，用于在不改核心订单、租户、账本和权限边界的前提下扩展。长期目标接近 WordPress 插件的业务扩展体验：开发者可以围绕平台提供的稳定扩展点开发新功能，例如支付、外部货源、TRX 兑换、通知、运营工具或后台面板，而不是直接修改核心订单和租户代码。该目标态不是当前交付范围；当前阶段只实现受控适配器插件的边界和最小合同。

- 支付通道，例如后续增加更多易支付变体、数字货币网关或线下收款适配。
- 外部发卡源和供货代理，例如对接 acg-faka、mcy-shop 等上游程序的商品目录、下单、查单和取发货。
- 可选 Bot/Web 管理功能，例如客服、文件处理、通知、运营辅助工具、TRX 兑换等租户工具。

插件不是独立绕过平台的业务系统。目标态插件必须经过平台注册、能力声明、租户启用、权限校验、凭据加密、审计和离线合同测试；当前代码暂未实现数据库级插件安装、租户级插件启停或在线升级。

## 2. 阶段范围

插件系统分两层推进：

1. **当前阶段落地受控适配器插件。** 只把 `payment` 和 `external_source` 作为近期可执行主线，新增 `app/services/business_plugins.py` 中的最小 manifest、能力摘要和静态 registry 合同，用于把现有支付 provider、外部源 provider 汇总成业务插件视角。该层不做在线安装、不做远程代码执行、不做真实第三方联调。
2. **目标态预留平台扩展插件。** `bot_feature`、`admin_web_panel`、`background_job`、`webhook_handler`、`tenant_tool` 等作为 manifest kind 和架构边界预留，后续可以支持类似 TRX 兑换这类新业务功能，但第一阶段不实现动态 Bot router 注入、任意后台任务注入、远程前端 JS 或插件市场。

这样可以现在固定扩展边界，避免后期大面积改核心业务；同时不把当前 Admin Web 和真实支付/外部源联调主线拖成完整插件平台建设。

### 2.1 当前架构决策记录（2026-06-14）

本轮确认：更宽定义的 WordPress 式插件能力有长期价值，但当前不进入实现。当前必须先做并持续补齐的是受控 `payment` / `external_source` 适配器方案，以及能支撑它们的最小插件合同。

原因如下：

- 当前最容易导致后期大改的是支付、外部货源、商品映射、付款后代发、凭据和订单履约边界；这些问题用 provider protocol、manifest、租户连接、签名 handle、安全 BFF 和离线合同即可锁住。
- WordPress 式通用插件运行时还会引入插件安装升级、插件数据库迁移、动态后台菜单、动态前端组件、Bot handler 注入、后台任务注入、Webhook handler 注入、权限沙箱和版本依赖解析。现在强行实现会明显扩大主线范围，并增加资金、发货和租户隔离风险。
- TRX 兑换、通知工具、运营工具等未来功能可以先以 `tenant_tool`、`background_job`、`webhook_handler` 等 manifest kind 保留语义，但只有在出现明确业务需求、稳定权限模型和离线合同后，才新增显式核心扩展点。

因此当前代码演进规则是：

1. 新增支付系统时，只允许先走 `PaymentProvider`、支付配置 schema、`payment` manifest、脱敏 BFF、OpenAPI 安全合同和离线测试。
2. 新增 acg-faka、mcy-shop 或其他发卡上游时，只允许先走 `ExternalSourceProvider`、外部源连接凭据 schema、目录同步、外部下单、取发货、attempt 审计、脱敏 BFF 和离线合同。
3. Admin Web 只能渲染核心代码认识的配置表单和安全摘要；不加载插件提供的远程 JS，不让插件直接注册任意页面。
4. Bot、Public Store、OrderService、LedgerService 不直接 import 具体插件实现，只依赖核心协议和服务层编排。
5. 在出现至少两个非支付/非外部源的真实插件需求并形成重复扩展点前，不建设通用插件运行时。

## 3. 当前判断

项目可以支持插件化接入 acg-faka、mcy-shop 这类发卡程序的商品，但不应把它们当作普通供应商店铺直接读写。

当前不建议立刻建设 WordPress 式通用插件运行时。原因是通用插件会同时改变安装升级、数据库迁移、菜单注入、Bot router 注入、后台任务、Webhook handler、前端面板和权限沙箱，范围明显超过当前 Admin Web、支付和外部货源主线。现在必须做的是冻结支付/货源适配器边界，并要求后续新增支付或外部货源只能通过 provider protocol、manifest registry、租户连接和安全 BFF 接入；这样能避免项目完成后再接 acg-faka、mcy-shop 或新支付系统时大改订单、商品、账本和租户隔离代码。

推荐模型是“外部货源插件 + 本地代理商品”：

1. 租户在后台创建外部源连接，凭据加密保存。
2. 插件同步上游商品目录到本地商品映射，只保存安全摘要和外部三元组：`provider_name/source_key/external_product_id`。
3. 租户选择外部商品后，以本地自营商品或供货商品形式上架；买家侧只看到本店商品，不暴露上游平台身份和商品 ID。
4. 买家付款成功后，后台外部履约 worker 使用同一 `out_trade_no` 向上游下单，随后拉取发货内容并导入本地待发送发货记录。
5. 本地 Bot/Public Store 继续负责最终发货、订单状态、售后、争议、账本和风控。

这样能实现“有人购买时才自动从上家渠道购买并发给购买人”的代发逻辑，同时保留本平台的多租户隔离、支付收款、审计和风险控制。

### 3.1 当前必须冻结的代码边界

- 核心订单、商品、账本和租户服务只依赖 `PaymentProvider`、`ExternalSourceProvider`、provider registry 和业务插件 manifest 摘要，不直接 import 具体第三方项目适配器。
- Admin Web、Tenant Admin API、Public Store 和 Bot 不允许浏览器或买家请求传入内部 `tenant_id`、外部源连接 ID、支付配置 ID 或上游商品/订单内部字段；租户和连接必须由当前会话、API Key、订单或签名 handle 在服务端解析。
- 新增支付通道必须先进入 `payment` manifest 和支付 provider 能力摘要，再接租户/平台配置、离线合同、OpenAPI 安全响应和 readiness marker。
- 新增外部货源必须先进入 `external_source` provider protocol、连接凭据 schema、目录同步/下单/取发货离线合同和脱敏测试，再考虑真实 staging。
- 所有真实第三方请求必须从服务层受控发起，不允许插件在 import、manifest 解析、Admin Web 能力摘要或列表页加载时访问网络。

### 3.2 当前明确不做的通用插件能力

- 不做在线安装、卸载、升级和插件市场。
- 不做插件自带 Alembic 迁移、动态建表或任意 SQL 执行。
- 不做远程前端 JS、动态 React 面板注入或任意菜单注册。
- 不做动态 Bot router/handler 注入、任意后台任务注入或任意 Webhook handler 注入。
- 不做插件级 Python 沙箱承诺；第一阶段插件代码仍是平台受信任代码或离线 fixture。
- 不把 TRX 兑换、运营工具、通知工具等 `tenant_tool` 当作当前交付，只在 manifest kind 和文档中保留目标态扩展点。

## 4. 已有基础

当前代码已经具备受控适配器插件的基础切入点，但尚未具备完整插件包运行时：

- `app/services/business_plugins.py`：定义业务插件 manifest、插件类型、能力声明、静态 registry，以及将现有支付/外部源 provider 摘要转换为插件 manifest 的最小合同。
- `app/services/external_sources/base.py`：定义外部目录、外部订单和外部发货 Provider Protocol。
- `app/services/external_sources/registry.py`：提供 provider 注册、查询和能力摘要。
- `app/services/external_sources/connections.py`：提供租户级外部源连接、凭据加密保存、脱敏摘要和运行时凭据加载。
- `app/services/external_sources/sync.py`：提供外部目录同步到本地商品元数据。
- `app/services/external_sources/orders.py`：提供外部下单、查单和取发货编排。
- `app/services/external_sources/auto_fulfillment.py`：提供付款后外部自动履约尝试和 attempt 记录。
- `app/services/payments/base.py`：定义支付 Provider Protocol。
- `app/services/payments/configs.py`：提供支付 provider 能力摘要和租户/平台支付配置解析。

当前不足：

- provider 仍是内置注册或静态汇总，不支持插件包安装、版本依赖解析、数据库级启停、租户级插件配置表或在线升级。
- `app/services/business_plugins.py` 只做 manifest/registry 合同，不导入执行第三方插件代码，不触发 provider 方法，不发外部网络请求。
- `mcy_shop` 目前只是离线 fixture skeleton，不是真实 mcy-shop 适配器。
- 尚未实现 acg-faka 真实适配器。
- 外部自动履约当前运行时只按 provider registry 摘要检查 context 型订单/发货能力和 provider 对象 `auto_fulfillment_idempotent=True`；manifest 门禁、租户级插件启停和真实 staging 幂等证明属于后续接真实插件前的目标约束。
- Admin Web 插件能力摘要只读 BFF 已接入：`GET /api/v1/admin-web/business-plugins/capabilities` 只从 httpOnly Admin Web 会话的当前工作区解析平台或克隆 Bot 上下文，返回 `list_current_business_plugin_manifests()` 生成的浏览器安全投影，以及当前工作区的非密文支付启用状态和外部源连接计数。该接口不执行插件 entrypoint，不导入远程代码，不调用 provider 下单/目录/发货方法，不读取或解密外部源凭据，不读取支付密钥，不测试上游连通性，不返回 `tenant_id`、连接 ID、凭据字段、raw payload、外部订单 ID、发货内容或 storage key；这不代表插件安装、租户级启停、真实 mcy-shop/acg-faka、真实支付网关或 staging 验证完成。
- Admin Web 外部源连接管理合同已接入：`GET /api/v1/admin-web/tenant/external-source-connections`、`POST /api/v1/admin-web/tenant/external-source-connections` 和 `POST /api/v1/admin-web/tenant/external-source-connections/disable` 只从 httpOnly Admin Web 当前克隆 Bot 工作区解析租户，列表只返回 provider 能力摘要、连接展示名、状态、时间、`credential_field_count` 和签名 `connection_handle`，创建只做本地 provider 凭据格式校验和加密保存，停用只按 `connection_handle` 定位当前租户连接。该入口不读取或解密外部源凭据，不调用 provider 目录/下单/发货方法，不触发真实上游联调，也不代表 acg-faka/mcy-shop 真实连接已认证。
- Admin Web 外部源目录同步合同已接入：`POST /api/v1/admin-web/tenant/external-sources/catalog/sync` 只从 httpOnly cookie 当前克隆 Bot 工作区和签名 `connection_handle` 解析租户与 active 外部源连接，请求仅允许 `connection_handle`、`cursor`、`limit`、`max_pages`。服务端受控加载 runtime credentials 后复用 `ExternalCatalogSyncService.sync_registered_catalog()` 同步一页或少量分页目录，响应只返回 provider/source、同步计数、next cursor 和本地商品安全摘要，不返回 `tenant_id`、`connection_id`、`external_id`、`raw_payload`、`credentials`、`token`、`secret`、`storage_key`、`delivery` 或发货内容。该入口是离线/受控 BFF 合同，不代表真实 acg-faka/mcy-shop 上游已经联调。

## 5. 插件类型

### 5.1 payment

用于新增支付通道。插件必须实现或适配：

- 创建支付订单。
- 回调验签和 payload 归一化。
- 可选查单和对账。
- provider 能力摘要。
- 租户级或平台级配置 schema。
- 敏感配置字段脱敏。

近期主线仍只聚焦 `epusdt_gmpay` 和 `epay_compatible` 的配置入口、离线合同和后续 staging 验证。其他支付插件可以先进入 `offline_only` 或 `staging_verified=false` 状态，不进入生产默认通道。

### 5.2 external_source

用于接入外部发卡源和代发上游。插件必须按能力实现：

- 商品目录列表和单品查询。
- 创建外部订单。
- 查询外部订单。
- 拉取外部发货。
- 凭据 schema 和本地校验。
- 错误分类、可重试判断和敏感信息脱敏。
- 是否声明同一 `out_trade_no` 幂等自动履约。

acg-faka 和 mcy-shop 应优先作为 `external_source` 插件，而不是硬编码进订单服务。

### 5.3 bot_feature

用于后续扩展 Bot 功能，例如自动客服、通知、文件处理等。此类插件必须挂在租户功能开关下，并通过现有租户上下文、RBAC、审计和限流。

此类型不属于第一阶段交付。第一阶段仅保留 manifest kind 和安全边界，不实现运行时动态 Bot router 注入；后续也只允许核心代码显式加载已安装且通过 allowlist 的插件。

### 5.4 admin_web_panel

用于给 Web 管理后台增加配置或观测面板。第一阶段仅允许插件提供 metadata，由核心前端按已知组件渲染配置表单；不执行远程 JS，不加载任意前端代码。

此类型不属于第一阶段交付。第一阶段最多展示核心代码生成的插件能力摘要，不执行远程 JS，不加载任意前端代码。

### 5.5 tenant_tool / background_job / webhook_handler

用于目标态业务扩展，例如 TRX 兑换、链上观察、定时汇率同步、外部通知回调等。

此类插件必须满足：

- 所有租户写操作从服务端 session/API Key/worker 查询出的租户上下文推导，不接受浏览器传入内部 `tenant_id`。
- 后台任务由平台调度器显式注册和限流，不能由插件 import 阶段启动线程或进程。
- Webhook handler 必须有独立验签、大小限制、字段白名单和审计。
- 涉及资金、兑换、链上或真实第三方请求时，必须先完成离线合同和 staging 验证。

第一阶段只保留 kind，不实现动态任务和 handler 注入。

## 6. 插件包结构

目标态插件包建议使用独立 Python 包或仓库目录，结构如下。当前代码只校验 manifest 和 entrypoint 字符串，不会扫描、安装、import 或执行这些插件包：

```text
plugins/
  fakabot_ext_acg_faka/
    fakabot_plugin.json
    __init__.py
    provider.py
    schemas.py
    tests/
```

`fakabot_plugin.json` 示例：

```json
{
  "plugin_id": "acg_faka",
  "name": "acg-faka 外部货源",
  "version": "0.1.0",
  "kind": "external_source",
  "entrypoint": "fakabot_ext_acg_faka:create_provider",
  "contract_version": "external_source_v1",
  "capabilities": {
    "catalog": true,
    "order": true,
    "delivery": true,
    "auto_fulfillment_idempotent": false
  },
  "production_ready": false,
  "staging_verified": false
}
```

插件加载规则：

- `plugin_id` 必须是小写字母、数字、下划线和短横线。
- `entrypoint` 必须是 `module:function` 格式，且只能从平台 allowlist 路径加载。
- 插件不能在 import 阶段发网络请求、读 `.env`、启动线程或执行迁移。
- 插件必须显式声明 `production_ready` 和 `staging_verified`。
- 插件只返回 provider 对象和 schema，不直接访问数据库会话。
- 当前代码只实现 manifest 校验、entrypoint allowlist 判定和静态 registry，不会自动 import 或执行插件 entrypoint。

## 7. 核心边界

### 7.1 租户隔离

插件不能接收或信任浏览器传入的 `tenant_id`。租户身份只能来自：

- Tenant Admin API 当前 API Key。
- Admin Web 当前 httpOnly session cookie 工作区。
- 后台 worker 查询出的本地订单和商品。

### 7.2 凭据安全

插件凭据通过 `ExternalSourceConnectionService` 或支付配置服务加密保存。插件运行时只能收到解密后的 `runtime_auth`，且：

- `repr()` 必须脱敏。
- HTTP 响应和审计不得回显凭据。
- 上游错误不得原文返回浏览器或公开 API。
- OpenAPI schema 不出现 `api_key`、`secret`、`token`、`credential`、`storage_key` 等敏感响应字段。

普通 `ExternalSourceOperationContext` 和 `ExternalCatalogSyncContext` 只承载租户、provider、source_key、connection_id 等选择信息，不承载凭据；服务层校验 active、本租户、未删除连接后，才允许用 `ExternalAuthenticatedOperationContext` / `ExternalAuthenticatedCatalogSyncContext` 在内存中短暂携带 `runtime_auth`，其 `repr()` 必须脱敏。

### 7.3 raw payload 边界

Provider DTO 内部可以短暂携带 `raw_payload` / `raw_response` 供服务层准入校验、失败分类或脱敏审计使用，但 HTTP 响应、Public Store、Bot、Admin Web 安全 DTO 和 OpenAPI schema 不得返回上游 raw payload、发货 `items/message`、凭据字段、密文、完整外部订单上下文或 storage key。

### 7.4 自动履约

插件只有在满足以下条件时才能进入后台自动履约：

- provider 实现 context 型 `create_order` 和 `fetch_delivery`。
- provider 对象声明 `auto_fulfillment_idempotent=true`，且 registry 摘要 `auto_fulfillment_idempotent_available=true`。
- 对于后续动态插件，manifest 也必须声明 `auto_fulfillment_idempotent=true`。
- 已通过离线重复 `out_trade_no` 回放测试。
- 接真实第三方前已完成真实 staging 幂等验证并记录文档。

未满足条件时，插件仍可用于目录同步和人工操作入口，但后台 worker 不自动调用。当前代码还没有把 manifest、租户级插件启停或 staging 证明接入 worker 运行时门禁，这些属于真实插件化前必须补齐的后续任务。

### 7.5 供应商身份保护

通过外部源插件代发时，买家侧、Public Store、租户 Bot 和代理商品公开响应都不得暴露：

- 上游平台名称。
- 上游店铺或供应商身份。
- 外部商品 ID。
- 外部订单 ID。
- 外部发货 raw payload。
- 外部源连接 ID 或凭据字段。

## 8. acg-faka 和 mcy-shop 接入路线

公开项目页面显示 acg-faka、mcy-shop 属于带插件/货源/共享店铺能力的发卡程序，适合作为外部货源插件候选。真实接入前仍需以对应版本的官方接口、数据库结构或插件文档为准，不能从 README 推断生产协议。

### 8.1 阶段 A：离线协议梳理

- 固定目标版本和部署方式。
- 确认是否有官方 HTTP API、插件接口或共享店铺协议。
- 列出商品目录、库存/价格、下单、查单、取发货、退款/失败状态字段。
- 确认鉴权方式、签名算法、幂等键、错误码和限流策略。
- 明确哪些商品类型可导入：首期只允许文本卡密或安全文本发货，不导入上游文件 storage key。

### 8.3 阶段 C：Admin Web 接入

- 第一阶段已先完成只读插件能力摘要：核心 BFF 和 `web/admin` 右侧“插件能力”卡片展示 payment/external_source 静态 manifest、生产/联调标记、离线状态、租户配置状态和连接计数；页面独立刷新、加载、错误和空态均不影响商品、订单、支付配置或供货面板。

## 9. 数据库建议

已有 `external_source_connections`、商品外部映射字段和 `external_fulfillment_attempts` 可继续复用。

新增插件管理建议表：

- `plugins`：安装状态、版本、kind、entrypoint、contract_version、production_ready、staging_verified。
- `plugin_tenant_settings`：租户启用状态、功能开关、展示配置。
- `plugin_audit_logs` 或复用 `audit_logs`：记录安装、启用、禁用、配置变更。

第一阶段可以先不新增表，用 `app/services/business_plugins.py` 的静态 registry + manifest 合同验证插件接口；进入可安装插件、租户级启停或平台级插件管理时再迁移。

## 10. 验收标准

- 当前阶段：业务插件 manifest 能校验 ID、类型、版本、入口、合同版本、能力、`production_ready` 和 `staging_verified`，并能把现有 payment/external_source provider 摘要转换成插件摘要。
- 当前阶段：entrypoint 只做 allowlist 判定，不自动 import、不执行第三方代码、不触发外部网络。
- 目标阶段：插件 registry 能列出插件、能力、版本和生产状态。
- 目标阶段：禁用插件不会出现在租户配置页，也不会被 worker 调用。
- 插件导入失败不影响核心订单、支付、账本。
- 外部源插件公开响应不暴露凭据、发货内容、raw payload、外部订单 ID 或内部租户 ID。
- 自动履约必须有幂等证明；没有证明的插件只能人工触发或只做目录同步。
- readiness 脚本新增插件合同 marker，缺失安全测试时失败。

## 11. 当前决策口径

当前不把“类 WordPress 插件运行时”作为近期主线实现，但必须现在保留并固定扩展边界。近期只做 `payment` 和 `external_source` 两类受控适配器插件，以及支撑它们的 manifest、provider protocol、registry、安全 BFF、租户连接、凭据加密、脱敏响应、离线合同和 readiness marker。

这个取舍的核心原因是：完整通用插件运行时会同时牵涉安装/卸载/升级、数据库迁移、菜单和后台面板注入、Bot handler 注入、后台任务调度、Webhook handler、权限沙箱、版本兼容和回滚机制，风险和工作量会压过当前 Admin Web、支付和外部货源主线。现在过早实现会让核心订单、租户、账本和权限边界被不稳定的插件生命周期牵着走。

但这不等于把插件能力放到项目完成后再补。当前必须坚持的代码约束是：新增支付系统和新增货源系统不得硬编码进订单、商品、账本、Public Store 或 Bot handler；它们只能通过 provider protocol、manifest registry、租户配置/连接、服务层编排和安全 DTO 接入。只要这一层保持稳定，后续接入 acg-faka、mcy-shop 或更多支付系统不会要求大面积重写核心业务。

宽定义插件能力可以作为目标态继续演进。等至少出现两个以上明确的非支付/非货源插件需求，例如 TRX 兑换、独立后台工具、链上监听或通知工作流，再引入可安装插件表、租户级插件启停、受控任务注册、受控 Webhook 注册和核心前端渲染的配置面板；仍不应直接开放远程 JS、任意 SQL、任意 Bot router 或 import 阶段副作用。

因此当前推荐结论是：**先做“支付/货源适配器插件”方案，文档和核心边界按宽定义插件预留；不现在建设 WordPress 式完整插件运行时。**

## 12. 近期不做

- 不做任意第三方插件热加载和远程代码执行。
- 不做插件市场、在线安装和自动更新。
- 不做动态 Bot router 注入、动态后台任务注入或远程 Admin Web JS。
- 不做真实 acg-faka/mcy-shop 联调，除非用户明确确认测试环境和凭据。
- 不把更多支付通道拉回近期主线；支付近期仍聚焦 EPUSDT 和易支付兼容。
