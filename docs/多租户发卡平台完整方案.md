# Telegram 多租户发卡 Bot 克隆平台完整方案

## 1. 文档目标

本文档基于 `docs/需求初版.md`，将“多租户 Bot 克隆 + 发卡业务 + 自营/供货代理 + 平台收费 + 外部接口预留”的需求整理成可实施的完整方案。

本文档描述目标架构、落地路径和当前新架构进展。旧版单 Bot 代码仍保留为业务参考；`app/` 新架构已经具备多租户、订单、支付、外部源、Web 管理后台等离线合同和部分本地闭环，但不代表真实 Telegram、真实支付、真实数据库迁移或真实第三方外部发卡源已经完成生产验证。

## 2. 当前项目现状

当前代码主要特点：

- 技术栈：`python-telegram-bot + Flask + SQLite + Redis`。
- 运行模型：单个 `BOT_TOKEN` 初始化一个 Telegram Bot。
- 数据存储：本地 SQLite，核心表包括商品、订单、卡密、设置、USDT 直付记录等。
- 支付能力：旧版代码已有易支付/柠檬支付、TOKEN188、TRC20-USDT 直付扫链相关实现；新架构已迁移部分离线合同、配置和本地支付/回调编排，但生产可用性仍以真实 staging 验证为准。
- 发货能力：已有自动拉群、通用卡密/文本、卡池文本发货；未发现 zip/rar/7z 文件商品、`.session`、`tdata` 或文件夹打包售卖的专用上传/交付逻辑。
- 部署方式：已有 Dockerfile 和 docker-compose，Redis 已纳入编排。

与目标需求的差异：

- 目标要求 `aiogram 3.x + PostgreSQL + Redis`。
- 目标要求单进程多租户，多 Token Webhook 动态分发。
- 目标要求租户数据强隔离、Token 加密、平台账本、供货代理和外部 API。
- 当前 `bot.py`、`user_flow.py`、`admin_panel.py` 文件较大，不适合直接在原结构上继续叠加多租户复杂度。

结论：建议重建平台内核，迁移现有发卡能力，而不是在当前大文件里硬改多租户。

## 3. 产品定位

平台定位为 Telegram Bot-as-a-Service 发卡 SaaS：

- 平台运营者维护一套后端服务和母 Bot。
- 商家通过母 Bot 绑定自己在 BotFather 创建的 Bot Token，获得一个独立发卡 Bot。
- 每个克隆 Bot 是一个租户，拥有独立商品、订单、用户、支付设置、Bot 文案和功能开关。
- 商家可以只卖自营商品，也可以开放供货，或代理其他供货商商品。
- 开启供货/代理等涉及分账的功能后，订单必须走平台收款，由平台账本记录各方余额，再人工提现审核。

## 4. 第一版范围

第一版必须完成：

- 母 Bot 引导绑定 Token、设置 Webhook、管理克隆 Bot。
- 多 Token Webhook 路由。
- 租户识别、中间件注入当前租户上下文。
- 租户级商品、卡密、订单、设置、支付方式。
- 自营发卡闭环。
- 平台订阅收费：首月免费，第二个月起默认每月 10 USDT，可由平台后台调整。
- epusdt 支付适配器。
- 基础 Docker Compose：应用、PostgreSQL、Redis。
- 基础安全：Token 加密、日志脱敏、租户权限校验、回调幂等。
- 租户管理员角色和基础 RBAC。
- 基础账单、订单报表导出。

第一版建议暂缓：

- 多级佣金。
- 自动链上提现。
- 复杂代理等级体系。
- 复杂运营 BI、跨平台客服工单和非 Telegram 登录体系。
- 外部发卡系统的深度双向同步。

第二版再完成：

- 供货市场。
- 代理上架。
- 供应商定价规则。
- 平台账本和人工提现审核。
- 商品审核、投诉、退款、订单争议。
- Cloudflare Workers 网页发卡 API。
- acg-faka、mcy-shop 等外部发卡源适配。

近期新增独立里程碑：

- 使用 `shadcn/ui` 开发 Web 管理后台，分为主 Bot 管理和克隆 Bot 管理设置。
- Web 后台必须支持绑定或选择目标克隆 Bot；代理商在网页供货市场选择商品并上架前，必须明确商品将代理售卖到哪个发卡 Bot。
- 支付系统近期只重点推进 `epusdt_gmpay` 和 `epay_compatible`/易支付兼容的配置入口、离线合同和 staging 验证路径，其他支付系统对接后置。

这样可以先验证多租户内核和自营发卡闭环，避免第一版范围过大。

## 5. 核心架构决策

### 5.1 技术栈

推荐目标技术栈：

- Python 3.11+
- aiogram 3.x
- FastAPI 或 aiohttp 作为 Webhook HTTP 服务
- PostgreSQL 15+
- Redis 7+
- SQLAlchemy 2.x async 或 SQLModel
- Alembic 数据库迁移
- Pydantic Settings 配置管理
- Docker + docker-compose

选择理由：

- aiogram 3.x 对异步、多 Router、Middleware 更友好。
- PostgreSQL 支持事务、行级锁、复杂索引和账本一致性。
- Redis 适合租户配置缓存、FSM 状态、限流、幂等锁。
- Alembic 保证表结构可迁移，避免手写 `ALTER TABLE` 扩散。

### 5.2 租户单位

租户单位定义为“一个克隆 Telegram Bot”。

每个租户包含：

- 一个 Bot Token。
- 一个租户 owner。
- 一套商品、订单、卡密、设置、菜单、支付配置。
- 一套自营/供货/代理功能开关。

平台母 Bot 本身不是普通租户，但可以复用部分管理能力。

### 5.3 Token 来源

第一版使用用户从 BotFather 手动创建的 Bot Token。

后续可评估 Telegram Managed Bot 能力，但不能阻塞 MVP。

### 5.4 Token 存储

Bot Token 必须加密存储：

- 数据库只保存 `encrypted_token` 和 `token_hash`。
- `token_hash` 用于去重、查找和审计，不可反推出原 Token。
- 原始 Token 只在绑定、设置 Webhook、运行时 Bot 实例初始化时短暂出现在内存中。
- 日志、异常、后台页面、通知消息中永远不输出完整 Token。

建议：

- 使用 Fernet 或等价 AES-GCM。
- 加密密钥通过环境变量提供，例如 `TOKEN_ENCRYPTION_KEY`。
- 支持 Token 轮换：更新密文、重设 Webhook、清理旧缓存。

### 5.5 Webhook 路由

不建议把原始 Bot Token 放在 Webhook URL。

推荐路径：

```text
POST /telegram/webhook/{webhook_secret}
```

处理流程：

1. 从 URL 中读取 `webhook_secret`。
2. Redis 缓存或 PostgreSQL 查询 `tenant_bots.webhook_secret`。
3. 找到租户和 Bot Token。
4. 解密 Token，获取或创建 aiogram `Bot` 实例。
5. 将 Update 喂给统一 Dispatcher。
6. TenantMiddleware 注入 `tenant_id`、`bot_id`、`owner_id`、功能开关和设置。

优点：

- URL 不暴露 Token。
- Webhook secret 泄露后可以单租户重置。
- 方便 Nginx、日志和监控脱敏。

### 5.6 Router 分层

Router 应按平台能力和业务能力分离：

```text
app/
  main.py
  config.py
  web/
    webhook.py
    health.py
    callbacks.py
  bots/
    factory.py
    dispatcher.py
    middlewares/
      tenant_context.py
      auth.py
      rate_limit.py
      language.py
    routers/
      master/
        start.py
        clone.py
        tenant_manage.py
        billing.py
      tenant/
        start.py
        catalog.py
        order.py
        payment.py
        admin_menu.py
        product_manage.py
        card_manage.py
        supplier.py
        reseller.py
        withdrawal.py
  domains/
    tenants/
    products/
    inventory/
    orders/
    payments/
    billing/
    marketplace/
    ledger/
    external_api/
  db/
    models/
    repos/
    migrations/
  services/
    telegram_webhook.py
    token_crypto.py
    payment_router.py
    delivery.py
    ledger.py
    subscription.py
  workers/
    order_expire.py
    inventory_unlock.py
    subscription_notice.py
```

原则：

- 母 Bot 管理 Router 与租户 Bot 业务 Router 分离。
- 租户上下文由 Middleware 注入，不允许业务代码从全局变量读取当前租户。
- Repository 负责数据访问，Service 负责业务事务，Router 只处理交互。

## 6. 角色和权限

### 6.1 平台管理员

权限：

- 管理平台配置。
- 查看所有租户状态。
- 冻结/解冻租户。
- 审核商品。
- 处理投诉、退款、订单争议。
- 审核提现。
- 调整订阅价格和免费期。

### 6.2 租户 owner

权限：

- 管理自己的克隆 Bot。
- 配置欢迎语、菜单、功能开关、支付方式。
- 管理自营商品和卡密。
- 开启或关闭供货商能力。
- 开启或关闭代理能力。
- 查看自己的订单和账本。
- 发起提现。
- 决定自己的 Bot 是否允许继续克隆。

### 6.3 租户管理员

第一版需要支持租户管理员角色，但权限必须明确收敛，不能等同于 owner。

建议内置角色：

- `owner`：租户所有者，拥有全部权限。
- 第一版落地为 `admin` 角色 + 可编辑权限项。
- 后续可按 `product_admin`、`order_admin`、`finance_admin`、`support_admin` 拆分预设角色。

当前可编辑权限项：

- `settings`：店铺设置。
- `products`：商品、库存、文件和群邀请。
- `orders`：发货重试和支付补偿。
- `payments`：租户支付配置。
- `finance`：账本、提现和提现记录。
- `supply`：供货代理。
- `api_keys`：租户 API Key。
- `subscription`：订阅查看和续费。

权限命令：

- `/permissions`：owner 查看 admin 角色权限。
- `/set_permission 权限 | on/off`：owner 开启或关闭 admin 角色权限。

兼容策略：

- owner 始终拥有全部权限。
- admin 在没有显式权限记录时保持原有可管理行为。
- owner 显式关闭某权限后，admin 无法访问对应命令和菜单入口。

敏感操作仅允许 `owner` 执行：

- 绑定、轮换、删除 Bot Token。
- 添加、移除租户管理员。
- 修改管理员权限。
- 删除或停用租户 Bot。
- 修改提现地址。

权限应在 Middleware 或 Service 层统一校验，不能只依赖按钮是否展示。

### 6.4 普通买家

权限：

- 浏览商品。
- 下单支付。
- 查询订单。
- 接收自动发货内容。
- 发起投诉或售后请求。

## 7. 功能模块方案

### 7.1 母 Bot 克隆流程

流程：

1. 用户向母 Bot 发送 `/start`。
2. 母 Bot 展示平台说明、收费规则和风险提示。
3. 用户选择“创建/绑定机器人”。
4. 母 Bot 引导用户到 BotFather 创建 Bot，并粘贴 Token。
5. 平台调用 Telegram `getMe` 验证 Token。
6. 校验 Token 未被其他租户绑定。
7. 加密保存 Token，生成 webhook secret。
8. 调用 `setWebhook` 设置租户 Bot Webhook。
9. 创建租户默认配置。
10. 提示首月免费和到期时间。

Token 消息处理：

- 用户发送 Token 后，平台应尽量删除该消息。
- 若删除失败，提醒用户手动删除。
- 后台不记录原始 Token。

### 7.2 克隆 Bot 管理

母 Bot 提供：

- 查看已绑定 Bot 列表。
- 查看 Bot 状态。
- 重设 Webhook。
- 轮换 Token。
- 删除/停用 Bot。
- 查看订阅到期时间。
- 续费入口。

克隆 Bot 若开启“允许继续克隆”，也可以提供同样入口：

- 入口归属当前克隆 Bot。
- 新租户仍然由平台统一管理。
- 第一版不做多级佣金，因此上级 Bot 只作为入口来源记录，不产生收益。

### 7.3 租户设置

每个租户可配置：

- 欢迎语。
- 首页封面。
- 店铺名称。
- 商品分类。
- 菜单按钮。
- 客服联系方式。
- 支付方式。
- 自营功能开关。
- 供货商功能开关。
- 代理功能开关。
- 是否允许其他用户从当前 Bot 入口克隆。
- 订单超时时间。
- 发货提示文案。

建议命令式设置和菜单式设置都支持：

- 命令式适合高级用户，例如 `/setwelcome`。
- 菜单式适合普通商家，通过 inline keyboard 和分步输入完成。

### 7.4 自营发卡

自营商品由当前租户创建和维护。

商品类型：

- 卡密。
- 固定文本。
- Telegram 群邀请链接。
- 普通文件或文件链接。
- 外部发货源。

库存规则：

- 卡密必须归属 `tenant_id`。
- 下单后先锁定库存，支付超时释放。
- 支付成功后将库存标记为已使用。
- 发货操作必须幂等，重复回调不会重复发卡。

文件类商品规则：

- 单个上传文件最大 5MB。
- 文件按租户隔离存储。
- Docker 部署时必须挂载持久化目录，避免容器重启或重建后文件丢失。
- 文件商品第一版只建议支持普通数字文件，不建议把 Telegram 账号会话文件作为内置商品类型。
- zip/rar/7z 等压缩包可以作为普通文件商品支持，但需要遵守相同的大小、鉴权、持久化和风控规则。
- 上传入口只允许 zip/rar/7z 压缩包：非白名单扩展名或明确 MIME 冲突会在下载 Telegram 文件前拒绝，缺失 MIME 或通用二进制 MIME 仅在扩展名为 zip/rar/7z 时继续交给后续文件头和压缩包扫描处理；该校验不替代反病毒扫描。
- 文件扫描必须阻断 zip 路径穿越、Windows 绝对路径、Telegram `.session`/`.session-journal`、`tdata`/`key_data`/`auth_key` 会话凭据和 `.env`/`.env.*`、`.netrc`、`cookies.txt`、SSH 私钥、AWS/GCloud 凭据路径等常见被盗凭据线索；RAR/7Z 第一版只做文件头基础校验，深度扫描后续单独评估。

文件上传交互可参考 GAFBot 的体验，但实现方式要平台化：

1. 商家在商品管理中选择“文件发货”。
2. Bot 进入等待上传状态，只接收 document 文件。
3. 系统校验文件大小、扩展名、MIME、压缩包结构和租户权限。
4. 文件保存到持久化存储，数据库只记录 `storage_key` 和元数据。
5. 商品上架后，买家支付成功只获得短期下载链接或由 Bot 转发文件。
6. 处理过程要有进度提示、失败提示和返回商品页入口。
7. 上传失败、取消或超时后要清理临时文件。

如果后续需要对 zip 做服务端处理，应作为异步文件任务执行，不在 Telegram Update 处理链路中同步阻塞。

### 7.5 供货商功能

租户开启供货商功能后，可以将自己的商品开放给其他租户代理。

供货商可配置：

- 哪些商品开放代理。
- 建议零售价。
- 固定代理成本价规则。
- 是否限制最低售价。
- 是否需要平台审核后上架，默认不需要。
- 是否需要供应商单独批准某个代理商。

当前 Bot 命令闭环：

- `/open_supply 商品ID | 建议价 | 最低售价 | 是否审批` 开放供货。
- `/set_supply_approval 供货ID | on/off` 设置代理审批开关。
- `/reseller_applications [数量]` 查看待审核代理申请。
- `/approve_reseller 供货ID | 代理租户ID` 通过申请。
- `/reject_reseller 供货ID | 代理租户ID | 原因` 拒绝申请。
- `/set_reseller_rule 供货ID | 代理租户ID | 成本 | 最低售价` 设置单个代理商成本和最低售价。

平台仍保留事后风控能力：

- 平台管理员可以下架违规供货商品。
- Platform Admin 供货商品状态管控 API 已提供最小 HTTP 切片，支持平台列出供货商品并执行软下架/恢复。
- 平台管理员可以冻结供货商或代理商。
- 创建订单前按买家近期下单频率和同币种金额阈值做拦截，命中时写入平台风控审计。
- 投诉、退款、订单争议会进入平台处理流程。

成本价规则：

- 固定成本价。
- 针对单个代理商覆盖。
- 基于自营售价加/减固定金额、百分比和分组批量设置后续再实现。

示例：

```text
供应商自营价：100 USDT
默认代理成本：自营价 - 20% = 80 USDT
代理商 A 单独成本：75 USDT
最低零售价：95 USDT
```

### 7.6 代理商功能

代理商可以在供货市场选择商品代理销售。

代理商可配置：

- 是否上架某个代理商品。
- 自己的销售价。
- 商品展示文案覆盖。
- 分类和排序。
- 是否隐藏供应商信息，默认允许隐藏。

售价规则：

- 如果供应商设置最低售价，代理商售价不得低于最低售价。
- 如果供应商未设置最低售价，允许代理商低于建议零售价。
- 代理商利润 = 实际销售价 - 供应商成本价 - 平台手续费。
- 当前默认不收平台手续费，仅使用月租收费；平台手续费能力保留，未来可在月租和抽成之间二选一。
- 若未来启用平台手续费，默认费率为 1%，并支持平台管理员修改。

当前 Bot 命令闭环：

- `/supply_market [数量]` 浏览供货市场。
- `/apply_reseller 供货ID` 对需要审批的供货商品提交申请。
- `/my_reseller_applications [数量]` 查看申请状态。
- `/resell_offer 供货ID | 售价 | 展示名` 上架代理商品。
- `/reseller_products [数量]` 查看已上架代理商品。

订单归属：

- 销售租户：代理商 Bot。
- 供货租户：供应商 Bot。
- 买家看到的是代理商 Bot。
- 库存和发货由供应商商品驱动。

供应商隐藏规则：

- 买家侧 Bot 消息不能展示供应商身份。
- 代理商公开店铺页面不能展示供应商身份。
- Cloudflare Workers 公开 API 不能返回 `supplier_tenant_id`、供应商用户名、供应商 Bot 用户名、原始商品 ID。
- 平台管理员后台和内部审计日志可以查看真实供应商关系。

### 7.7 平台账本和提现

开启供货/代理后，订单必须使用平台收款。

账本原则：

- 资金流必须用不可变流水记录，不直接覆盖余额。
- 每笔订单生成多条账本分录。
- 退款、冲正、提现也生成反向分录。
- 可用余额、冻结余额由流水汇总或定期快照计算。
- 平台托管订单默认设置 7 天款项冻结期，平台管理员可修改。
- 冻结期内分成进入 `pending`，不能提现。
- 到达 `settlement_available_at` 后，后台任务将金额转入 `available`。
- 当前实现已提供只读账户余额审计，可按账本流水重算 pending/available/frozen 并和账户快照对比；已补基础数据库 `CHECK` 约束和结构化外部打款凭证字段。完整双边会计模型和已提现后退款追偿自动化后续单独补强。

示例：

```text
订单金额：120 USDT
供应商成本：80 USDT
代理商利润：35 USDT
平台手续费：0 USDT

账本：
- 平台收款 +120
- 供应商待结算 +80
- 代理商待结算 +35
- 平台服务收入 +0
```

若未来启用 1% 平台手续费，示例中的平台服务收入再按规则生成。

退款规则：

- 按各方实际获得金额等比例扣回。
- 供应商拿了多少，退款时扣回多少。
- 代理商拿了多少，退款时扣回多少。
- 平台手续费若已启用，也按实际收取金额扣回或按平台规则处理。
- 冻结期的目的就是确保退款期内仍有可扣回资金。

自收款限制：

- 只售卖自营商品且未开启供货/代理时，可以使用租户自己的收款地址。
- 自收款订单资金不经过平台，平台无法冻结、强制退款或从账本扣款。
- 自收款订单可以记录投诉和售后，但退款执行依赖租户自行处理。

提现流程：

1. 供应商或代理商查看账户余额。
2. 只能对 `available` 余额发起提现申请，填写地址和金额。
3. 系统将对应余额从 `available` 转入 `frozen`。
4. Bot 通知平台管理员处理。
5. 管理员人工转账后标记已完成，余额进入 `withdrawn`。
6. 若拒绝提现，释放冻结余额回 `available`。

第一版不建议做自动链上转账，避免资金安全和合规风险过高。

### 7.8 平台订阅收费

默认规则：

- 克隆成功后首月免费。
- 第二个月开始每月 10 USDT。
- 金额、免费天数、宽限期由平台后台配置。

需要支持：

- 到期前提醒。
- 到期后宽限期。
- 宽限期结束后冻结租户 Bot。
- 续费后恢复。
- 平台管理员手动赠送天数或调整到期时间。

冻结策略：

- 冻结后普通买家不能下单。
- 租户 owner 仍可进入续费和数据导出入口。
- 已支付未发货订单必须继续处理，不能因冻结丢失。
- 订阅欠费后数据保留 1 个月。
- 欠费保留期内，owner 可续费恢复或导出数据。
- 欠费超过保留期后进入待删除或归档流程，具体删除动作应由平台管理员确认。

## 8. 支付方案

### 8.1 支付模式

支付分两类：

1. 租户自收款。
2. 平台托管收款。

允许租户自收款的条件：

- 只开启自营。
- 未开启供货商功能。
- 未代理其他供应商商品。
- 未启用平台统一分账场景。

必须平台托管收款的条件：

- 订单包含代理商品。
- 租户开启供货/代理功能。
- 平台需要扣手续费、冻结保证金或处理分账。
- 订阅续费。

币种策略：

- 订单和账本的结算基准优先使用 USDT。
- 暂时保留 CNY 和其他币种字段，便于后续展示和扩展。
- 若后续接入 USDT/CNY 汇率 API，可支持按用户偏好切换显示。
- 历史订单必须保存下单时的汇率快照，避免汇率变化影响历史账务。
- 报表导出时同时输出原始币种金额、结算币种金额和汇率快照。

### 8.2 支付适配器

支付模块应使用统一接口：

```text
PaymentProvider.create_order(order, config) -> payment_url/payment_payload
PaymentProvider.verify_callback(request, config) -> callback_result
PaymentProvider.query_order(provider_trade_no, config) -> status
```

适配器：

- `epusdt_gmpay`：近期优先推进通道，用于平台托管收款、订阅续费和代理订单；当前能力摘要仍以真实 staging 验证为准。
- `epay_compatible`：近期第二推进通道，按易支付兼容主流程推进；真实查单、对账和生产验证仍后置。
- `usdt_trc20_direct`：只保留离线配置、付款说明和本地匹配骨架，真实扫链和生产对账后置。
- `token188`：保留现有离线签名/回调合同，真实网关联调和对账查询后置。
- `lemzf`：保留现有易支付兼容派生合同，不作为近期主支付通道。

### 8.3 epusdt 接入

新架构已新增 epusdt GMPay 适配器和本地基础闭环；以下字段和签名仍作为接入合同，真实网关、真实回调和 staging 验证需要在用户明确确认后执行。

epusdt 推荐接入 GMPay：

- 建单接口：`POST /payments/gmpay/v1/order/create-transaction`
- 常见字段：`pid`、`order_id`、`currency`、`token`、`network`、`amount`、`notify_url`、`signature`
- 成功返回中包含 `payment_url`
- 回调以 JSON POST 到 `notify_url`
- 回调验签后返回纯文本 `ok`

签名规则：

1. 保留所有非空字段。
2. 排除 `signature`。
3. 按 ASCII key 排序。
4. 拼接 `key=value&...`。
5. 末尾追加商户 `secret_key`。
6. 计算小写 MD5。

建议：

- epusdt 作为平台收款的优先实现。
- 租户自收款可选支持 epusdt，但要独立保存租户自己的商户配置。
- 支付回调必须通过 `provider + out_trade_no` 幂等处理。

### 8.4 回调幂等

必须保证：

- 同一个支付回调重复到达，只处理一次。
- 支付成功后发货只执行一次。
- 账本分录只生成一次。
- 状态流转必须有事务保护。

建议使用：

- `payment_callbacks` 表记录原始回调摘要和处理状态。
- `orders.out_trade_no` 唯一索引。
- `ledger_entries.idempotency_key` 唯一索引。
- Redis 短锁防止并发重复处理。
- PostgreSQL 事务和行级锁。

## 9. 外部接口和未来扩展

### 9.1 Cloudflare Workers 网页发卡

目标：

- 已有 Bot 的租户可以选择部署一个 Cloudflare Workers 网页前端。
- Workers 通过平台 API 获取商品、创建订单、查询支付状态。
- 发货仍由平台后端统一处理，避免把卡密和 Token 暴露给 Workers。

API 范围：

```text

GET  /api/v1/store/{tenant_public_id}/orders/{order_no}
POST /api/v1/store/{tenant_public_id}/orders/{order_no}/payment
```

安全要求：

- Public API 只返回可公开展示的数据。
- 创建订单和创建支付需要限流；第一版使用 Redis 分布式固定窗口限流，并在订单创建前接入买家频率/金额风险检测，检测阈值可通过 `ORDER_RISK_RECENT_WINDOW_SECONDS`、`ORDER_RISK_MAX_BUYER_ORDERS_PER_WINDOW`、`ORDER_RISK_DAILY_WINDOW_SECONDS`、`ORDER_RISK_MAX_BUYER_AMOUNT_PER_DAY` 配置。Public Store 的 POST 写接口应在租户查询和 Telegram WebApp initData 校验前先执行 client-only 限流和 Public Store IP allowlist，保护公开写入口成本；GET 公开读取接口不受该写限流影响，限流未命中时未知店铺仍返回 404。
- 订单风控拦截命中复用 `audit_logs` 写入 `platform_risk.order_creation_blocked`，只记录规则、阈值、窗口、买家 Telegram 用户 ID、金额、币种、租户和来源类型，不记录卡密、Token、支付密钥、payload 或文件存储键；Public Store 和租户 Bot 只提交该审计记录，普通下单失败不提交半成品订单状态。
- 订单风控自动处置默认关闭；开启后会按配置窗口和命中阈值累计 `platform_risk.order_creation_blocked`，达到阈值时自动写入 `platform_risk.user_auto_banned` 并复用平台用户封禁。公开响应仍只返回普通风控失败文案，不暴露自动处置阈值、内部规则或审计 metadata。
- WebApp initData 校验通过后，再按已验证业务主体计数：创建订单按 `buyer:{telegram_user_id}`，创建支付的订单维度限流必须在订单归属校验后执行，避免非订单买家消耗该订单的 subject 限流额度。
- Telegram WebApp 买家相关接口使用 `Telegram.WebApp.initData` 原始串验签，开启后从已验签 `user.id` 派生买家身份。
- 管理 API 使用租户 API Key，并可配置 IP/CIDR 白名单。
- Public Store 公开写接口可配置 IP/CIDR 白名单，默认关闭。
- 卡密永远不通过公开商品 API 返回。

- 外部目录同步服务只编排已注册 `ExternalCatalogProvider` 与本地商品仓储，按外部三元组同步商品元数据；服务层和 Tenant Admin API 均支持目录分页同步，也支持通过 `get_product` / `get_product_with_context` 同步单个外部商品；对支持上下文接口的 provider，可传入非敏感 `ExternalCatalogSyncContext`，字段限定为租户、provider、source_key 和 connection_id 等同步选择信息；连接只是 provider/source_key 选择器，同步过程不新增保存、不解密外部凭据，不覆盖库存，不删除商品，不创建外部订单，不拉取外部发货。

- 外部源连接服务提供运行时凭据 loader，用于后续真实 provider 在服务层受控解密：仅 active、本租户、未 deleted 连接允许解密；disabled 连接拒绝解密；缺失、跨租户或 deleted 连接返回空。运行时凭据对象只在内存中短暂持有明文，`repr`/`str` 必须输出脱敏摘要，不输出凭据明文、密文、敏感字段名或第三方认证 payload。
- Public Store 不返回外部商品身份字段。
- 外部源深度同步后续单独设计。

Tenant Admin 供货/代理商侧最小 API 要求：

Tenant Admin 供应商侧供货 API 要求：

- `GET /api/v1/tenant/supply/supplier-offers` 和 `GET /api/v1/tenant/supply/supplier-applications` 使用 `supply:read`。
- `POST /api/v1/tenant/supply/supplier-offers`、`PATCH /api/v1/tenant/supply/supplier-offers/{supplier_offer_id}/approval`、`POST /api/v1/tenant/supply/supplier-applications/approve`、`POST /api/v1/tenant/supply/supplier-applications/reject` 和 `POST /api/v1/tenant/supply/supplier-rules` 使用 `supply:write`。
- 所有入口只使用当前 API Key 的 `supplier_tenant_id`，不能接受请求体覆盖供应商租户。
- 创建供货时 `product_id` 只作为本租户自营商品输入，响应不返回 `product_id` 或 `variant_id`。
- 供应商侧允许返回 `reseller_tenant_id` 和 `reseller_store_name` 用于审核申请；不得返回 `supplier_tenant_id`、`supplier_store_name`、`rule_id`、库存/文件/凭据/密钥字段。
- 审批和拒绝只允许处理已有 `pending` 申请，不通过 HTTP 审批入口静默创建代理规则；平台审核仍后续。
- 独立代理规则最小 HTTP 切片已完成：`POST /api/v1/tenant/supply/supplier-rules` 只允许供应商按当前 API Key 为已有 `pending` 或 `active` 代理关系设置 `pricing_value` 和 `min_sale_price`，不创建不存在的代理规则，不返回 `rule_id`、供应商租户、底层商品/档位、库存/文件/凭据/密钥字段；它不是审批/拒绝接口的替代，平台审核和真实分账联调仍后续。
- Tenant Admin API 已同步接入功能开关业务防线：供应商侧接口进入服务前检查当前租户 `supplier` 开关，代理商侧接口进入服务前检查 `reseller` 开关；关闭时返回 403，不继续枚举市场、申请、供货商品或代理商品，也不写入供货/代理资源。

### 9.3 shadcn/ui Web 管理后台

目标：

- 提供比 Telegram 对话更适合长列表、筛选、批量查看和配置的管理后台。
- 分离主 Bot 管理和克隆 Bot 管理设置，避免平台运营权限和租户店铺权限混用。
- 让供应商和代理商可以在网页中浏览大量商品、筛选供货市场、配置代理售价，并明确绑定到某个克隆发卡 Bot。

技术口径：

- 新增 `web/admin/`，使用 Vite + React + TypeScript + Tailwind + `shadcn/ui`。
- 前端组件按 `.agents/skills/shadcn` 的规则开发：优先使用现有 shadcn 组件，使用语义色值，表单使用组件化 Field 结构，按钮和导航使用标准组件组合。
- 浏览器端不保存 Tenant Admin API Key、Bot Token、支付密钥、外部源凭据或卡密。
- 后端新增 Web session 适配层，使用 Telegram WebApp `initData`、平台管理员配置和租户成员关系计算权限。

工作区：

- 主 Bot 管理：租户/Bot 列表、Webhook 状态、订阅计划、租户订阅状态观测、支付通道观测、提现审核、平台风控、平台供货商品状态管控。
- 克隆 Bot 管理：店铺设置、商品、库存、订单、支付设置、订阅、财务、报表、API Key、审计日志、供应商和代理商功能。

Admin Web 平台租户订阅状态观测使用 `GET /api/v1/admin-web/platform/dashboard` 的安全摘要完成：平台管理员可查看 `trial_subscription_count`、`active_subscription_count`、`grace_subscription_count`、`suspended_subscription_count` 和 `retention_expired_subscription_count`，以及服务端 `subscription_attention` 关注队列中的宽限、暂停、保留期过期、已到期或临近到期租户。关注队列按严重程度和关键时间排序，只返回公开租户 ID、店铺名、owner Telegram 摘要、套餐/状态和关键时间。该能力只读，不暴露内部 `tenant_id`、订阅/套餐内部 ID、账单、支付流水、payload 或凭据，也不创建续费订单、不触发真实支付、Telegram 或生命周期 worker。

Admin Web 平台支付通道观测同样复用 `GET /api/v1/admin-web/platform/dashboard` 的安全摘要完成：平台管理员可查看 `payment_providers` 中各 provider 的能力、生产/联调/离线状态、支持资产/网络、租户配置数、启用数和配置缺口。该能力只聚合 `payment_provider_configs` 的非密文字段，不读取 `config_encrypted`，不解密租户支付配置，不返回网关 URL、商户号、密钥、收款地址、支付链接、上游流水、payload 或 raw 响应，也不调用真实支付网关。

Admin Web 商品批量状态更新使用 `PATCH /api/v1/admin-web/tenant/products/status` 完成：请求只允许 `product_ids` 和目标 `status`，服务端只从 httpOnly cookie 当前克隆 Bot 工作区解析租户，写操作必须通过 Origin 门禁，单次最多 50 个商品且不允许重复 ID。后端逐个复用自营商品状态更新校验，文件商品未绑定文件或群邀请商品未绑定群 ID 时整批拒绝；响应只返回更新数量和商品安全摘要，不返回内部租户 ID、库存项、文件 storage key、外部映射、凭据或 raw payload。`web/admin` 只提供当前页多选批量上架/下架，不做跨页隐式批量操作，也不触发真实 Telegram、支付或外部源调用。

绑定机器人规则：

- 从母 Bot 打开后台时，使用母 Bot Token 校验 `initData`，列出当前 Telegram 用户拥有或可管理的克隆 Bot。
- 从克隆 Bot 打开后台时，URL 携带安全的 `tenant_public_id` 或 `bot_public_id`，服务端用对应克隆 Bot Token 校验 `initData`，再校验 owner/admin 权限。
- 用户管理多个克隆 Bot 时，页面顶部必须显示当前 Bot 选择器。所有租户级写操作都按当前工作区执行，不能由请求体覆盖内部 `tenant_id`。
- 代理商选品、申请代理和上架代理商品前必须确认目标克隆 Bot；商品只上架到该 Bot 对应租户。
- 普通浏览器入口可后置实现一次性绑定码：Bot 里生成短期 code，Web 端用 code 换取服务端 session，code 存 Redis 并设置短 TTL。

供应商和代理商页面：

- 供应商页面支持选择本租户自营商品开放供货，设置成本、建议价、最低售价和审批策略，审核代理申请，并为单个代理商设置独立规则。
- 代理商页面支持浏览供货市场、搜索筛选、提交代理申请、设置展示名/售价/分类/排序和隐藏供应商，最终上架到当前克隆 Bot。
- 买家侧 Bot、Public Store 和代理商公开页面不得展示供应商内部身份、底层商品 ID、库存项、文件存储键、凭据或发货内容。

Tenant Admin API Key 网页管理合同：

- Admin Web API Key 管理合同已落地：`GET /api/v1/admin-web/tenant/api-keys`、`POST /api/v1/admin-web/tenant/api-keys` 和 `POST /api/v1/admin-web/tenant/api-keys/revoke` 只从 cookie 当前克隆 Bot 工作区解析租户，并复用租户 `settings` 权限；写操作必须通过 Origin 门禁，不使用 Tenant Admin API Key，不要求 HMAC 签名。
- 创建请求仅允许 `name`、`scopes` 和 `ip_allowlist`，明文 Key 只在创建响应中返回一次；列表和吊销使用签名 `credential_handle`，不返回 `api_key_id`、`tenant_id`、`key_hash`、`created_by_user_id`、内部用户 ID、Bot Token、支付密钥、payload、token 或 secret。
- `web/admin` 右侧“API Key”卡片已接入只读观测/完整租户管理预设、IP 白名单输入、独立刷新、创建后一次性明文展示和吊销确认；明文只保存在当前浏览器内存状态，刷新或切换工作区后不再展示。

Admin Web 报表任务下载代理合同：

- `GET /api/v1/admin-web/tenant/reports/export-jobs`、`POST /api/v1/admin-web/tenant/reports/export-jobs` 和 `POST /api/v1/admin-web/tenant/reports/export-jobs/download` 只从 cookie 当前克隆 Bot 工作区解析租户，不使用 Tenant Admin API Key，不要求 HMAC 签名；写/下载操作必须通过 Origin 门禁。
- 列表支持 `status`、`report_type` 和 `limit`，创建请求仅允许 `report_type`，下载请求仅允许签名 `download_handle`。列表/创建响应只返回报表类型、范围、状态、行数、是否可下载、签名下载句柄、脱敏失败原因和时间字段，不返回 `export_job_id`、`tenant_id`、`requested_by_user_id`、`filename`、`download_url`、`download_token`、`storage_key`、本地路径或原始错误。
- 下载入口只代理已完成且当前租户可下载的报表文件，下载文件名使用 `orders-report.csv` 等泛化名称；仍不生成报表、不启动 worker、不做真实环境下载联调。

Admin Web 订单观测合同：

- `GET /api/v1/admin-web/tenant/orders/observability` 只从 cookie 当前克隆 Bot 工作区解析租户，不使用 Tenant Admin API Key，不要求 HMAC 签名，读操作不要求 Origin 门禁。
- 查询仅支持 `limit` 和可选 `out_trade_no`，服务端复用支付回调失败、回调拒绝审计和外部履约 attempts 只读日志服务，返回支付回调失败、支付回调拒绝和外部履约 attempts 的安全摘要。

- `web/admin` 最近订单区域显示“订单观测”面板，覆盖支付回调失败、回调拒绝、外部履约 attempts 的空态、错误态和独立刷新态；单笔深挖仍走“订单排障”面板。

### 9.4 外部发卡系统适配

预留外部发卡源接口，用于后续接入 acg-faka、mcy-shop 等系统。

抽象接口：

```text
ExternalCatalogProvider.list_products()
ExternalCatalogProviderWithContext.list_products_with_context()
ExternalCatalogProvider.get_product()
ExternalOrderProvider.create_order()
ExternalOrderProviderWithContext.create_order_with_context()
ExternalOrderProvider.query_order()
ExternalOrderProviderWithContext.query_order_with_context()
ExternalDeliveryProvider.fetch_delivery()
ExternalDeliveryProviderWithContext.fetch_delivery_with_context()
```

第一版只保留数据结构和接口，不强行接具体系统。

当前实现落点：

- `app/services/external_sources/base.py`：外部商品、订单、发货数据结构，基础 Provider Protocol，以及目录/订单/发货可选的非敏感操作上下文协议。
- `app/services/external_sources/registry.py`：provider 注册、查询、列表和基于本地协议实现的能力摘要生成；registry 会校验 provider 对象的 `provider` 字段和调用方 `provider_name`，必须是非空字符串，trim 后只允许小写字母、数字、下划线和短横线，非法值按调用方参数错误处理；该规则只是本地标识符规则，不代表第三方平台账号名或真实连接状态。
- `app/services/external_sources/sync.py`：目录商品到本地商品元数据的幂等同步编排；服务入口会拒绝非法 `tenant_id`、`connection_id`、`source_key`、`cursor`、`limit`、`max_pages`、外部商品 ID 或 provider 协议对象，参数错误按调用方错误处理，不按外部源失败处理；目录分页同步通过 `list_products` / `list_products_with_context` 获取页面，单品同步通过 `get_product` / `get_product_with_context` 获取单个商品，两者复用同一套本地 upsert 和返回对象契约；provider 调用异常会包装为 `ExternalSourceError` 并由 Tenant Admin HTTP 返回 502。provider 返回的 `ExternalProductPage` 必须是有效分页对象，`products` 必须是 `ExternalProduct` 列表，`next_cursor` 必须是 `None` 或非空且不超过 512 字符的字符串，并且不能返回与当前请求相同的游标；页级契约错误按 provider 错误中断同步。`ExternalProduct` 必须匹配当前 provider，ID、名称、币种、发货类型、状态必须是字符串，外部商品 ID 最长 128 字符，价格必须是有限 `Decimal`，描述/分类必须是 `str | None`，库存数量必须是非负 `int | None`；这类 provider 契约错误中断同步。业务值不支持、外部商品不存在或 `ExternalProduct.raw_payload` 准入失败只跳过单个商品，`raw_payload` 必须是字典，递归字段名不得包含 token、secret、password、apikey、authorization、cookie、credential、session、plain_key、card_secret、storage_key 等敏感关键词。
- `app/services/external_sources/orders.py`：外部订单和发货 provider 最小编排，支持上下文型和旧版 provider 兼容调用；服务入口会规范化 `ExternalOrderRequest` 后再传给 provider，外部商品 ID 和可选文本字段会 trim，外部商品 ID 和外部订单 ID 最长 128 字符且不能含控制字符，`out_trade_no` 最长 96 字符，`buyer_reference` 最长 128 字符，`buyer_contact` 最长 256 字符；metadata 必须是字典、字段名 trim 后非空且不能重复，并拒绝递归敏感字段名，metadata 值会规范化为 JSON-like 数据，只允许 `None`、bool、整数、有限浮点数、字符串、数组和对象，字符串值最长 512 字符且不能含控制字符，数组最长 50 项，单层对象最多 50 个字段，字段名最长 128 字符，嵌套深度最多 4 层；这类请求参数错误按调用方错误处理。provider 主动抛出的 `ExternalSourceError` 原样透出，创建订单、查询订单或获取发货过程中的普通异常会包装为 `ExternalSourceError` 并保留异常链，Tenant Admin HTTP 入口按外部源失败返回 502。服务会对 provider 返回的 `ExternalOrder` / `ExternalDelivery` 做本地契约校验和归一化：订单 provider、订单 ID、商品 ID、状态、币种必须是字符串，数量必须是正整数，金额必须是有限非负 `Decimal`；发货 provider、订单 ID、发货类型必须是字符串，items 必须是 tuple/list 且每项为非空字符串，message 必须是 `str | None`；返回给上层前会 trim 文本字段并把发货条目固定为 tuple。`raw_payload` 必须是字典，递归字段名不得包含 token、secret、password、apikey、authorization、cookie、credential、session、plain_key、card_secret、storage_key 等敏感关键词。该能力只约束本地 provider 返回对象进入服务层的准入，不等同于完整日志脱敏；服务本身不直接写平台发货记录，平台发货落库由外部发货导入服务完成。
- `app/services/external_sources/fulfillment.py`：外部发货导入服务，仅支持已支付自营文本卡密商品；服务会按 `tenant_id + out_trade_no` 锁定本地订单，校验商品外部映射与 provider/source_key 一致，先按 `DeliveryRecord.order_id` 复用已有记录实现订单级幂等；无记录时把 `ExternalDelivery.items/message` 合并为一段文本，使用 `TOKEN_ENCRYPTION_KEY` 加密写入一次性 `InventoryItem(status=used, used_by_order_id=订单ID)`，再创建 `DeliveryRecord(status=pending)` 交给现有 Bot 发货链路。导入时不设置 `order.delivered_at`，发送成功后仍由 `mark_delivery_sent()` 标记；服务返回摘要包含 `imported/dry_run` 可观测字段，不包含卡密内容、外部 raw payload 或 runtime credentials。
- `app/services/external_sources/auto_fulfillment.py`：外部自动履约编排服务，用于已支付自营文本卡密外部映射商品。服务会先复用已有 `DeliveryRecord`，没有记录时按本租户 `provider/source_key` 加载 active 外部源连接 runtime credentials，调用已注册 provider 创建外部订单并拉取发货，再复用 `ExternalDeliveryImportService` 导入一次性本地已用库存和待发送发货记录。支付回调和支付对账不直接调用外部源，只把订单推进到 `paid`；`external_fulfillment` worker 默认每 `EXTERNAL_FULFILLMENT_INTERVAL_SECONDS` 秒扫描已支付、未发货的外部映射订单并尝试履约，失败会计入批次结果并写入 `external_fulfillment.failed` 审计，审计元数据不包含凭据、卡密、外部 raw payload 或敏感字段名。`delivery_dispatch` worker 默认每 `DELIVERY_DISPATCH_INTERVAL_SECONDS` 秒领取 `pending` 发货记录，复用现有 Bot 发货文本构造和 `mark_delivery_sent/failed` 状态机发送给买家；失败记录只标记 `failed`，不主动循环重试；每轮扫描前会按 `DELIVERY_SENDING_TIMEOUT_SECONDS` 把超时 `sending` 标记为 `failed`，供商家确认后手动重试，避免后台自动重复发货。当前没有外部订单持久化表，provider 适配器必须保证同一 `out_trade_no` 重试幂等，后续接真实第三方前应补适配器级验证或外部履约状态表。
- `app/services/external_sources/connections.py`：外部源连接加密保存、脱敏摘要和运行时凭据 loader；外部源标识符必须是字符串，且只允许小写字母、数字、下划线和短横线，非字符串或非法格式按调用方参数错误处理；凭据必须是非空 `dict[str, str]`，字段名和值 trim 后必须非空，字段名 trim 后不能重复，非 dict、非字符串字段名/值、空字段或重复字段按调用方参数错误处理；deleted 历史连接再次创建时复用原行重新激活，避免同一租户同一 provider/source_key 唯一约束冲突；loader 只作为服务层安全边界，已接入 Tenant Admin 目录同步和外部订单/发货入口，未传 `connection_id` 时不解密凭据，不发外部网络请求。
- Tenant Admin HTTP 层提供 provider 列表、外部源连接管理、受控目录同步编排、外部订单/发货最小操作入口和外部发货导入入口；provider 列表要求 `products:read`，只暴露本地能力布尔值，能力来自 provider 对象协议实现检查，不读取、不解密、不返回凭据，也不触发健康探测或外部网络请求；连接列表要求 `external_sources:read`，连接创建/禁用要求 `external_sources:write`，同步要求 `products:write`，外部订单创建要求 `external_sources:write`，外部订单查询和发货拉取要求 `external_sources:read`，外部发货导入要求 `external_sources:write`。目录同步和订单/发货入口均可通过 `connection_id` 引用已保存连接，并可把非敏感连接选择信息传给 provider；带连接时会受控解密运行时凭据并放入 repr 脱敏的 authenticated context，HTTP 响应不返回 provider `raw_payload`、凭据明文、密文或敏感字段名，provider 错误 502 统一返回泛化文案，不回显上游异常原文，发货导入响应显式返回 `imported/dry_run` 用于区分真实导入、dry-run 验证和已有发货记录复用，但不返回外部 `items/message` 或本地解密后的卡密内容，继续受 API Key、签名、限流和 IP 白名单约束。
- OpenAPI 安全契约测试已遍历所有 `/api/v1/tenant` operation，要求统一声明 Bearer/API-Key、签名 header 和 `x-fakabot-signature` 元数据；运行时拒绝路径已用 TestClient 覆盖缺少/无效 API Key、全局或单 Key IP 白名单拒绝、scope 不足、强制签名缺失/错误/过期和限流拒绝。当前测试使用 fake session/API Key，不连接真实数据库；API Key 生命周期已有服务级测试覆盖创建、认证、撤销、明文不落库、审计脱敏和租户隔离。
- Public Store OpenAPI 契约测试已遍历所有 `/api/v1/store` operation，确认不继承 Tenant Admin 鉴权或签名声明；公开创建订单请求和公开响应 schema 不暴露或接收租户内部 ID、库存 ID、支付配置、外部商品映射、凭据、卡密内容或存储路径。
- Public Store 商品列表转换测试已覆盖自营和代理商品脱敏；底层对象即使带供应商租户、外部三元组、库存明细或文件存储键，最终公开商品也只返回公开字段。公开商品 ID 默认使用租户级签名 ID，不再直接返回 `self:ID` / `reseller:ID` 内部数字形式；创建订单仍兼容旧 ID。
- Public Store WebApp 身份运行时拒绝路径已用 TestClient 覆盖：开启强制 initData 后，创建订单、查询订单和创建支付缺 initData 会在订单/支付/真实数据库业务流前返回 401；创建订单时 initData 无效或请求买家 ID 与已验签 `user.id` 不一致会返回 401；已验签用户跨买家查询订单或创建支付会返回 404，且不进入支付服务。
- Public Store 公开写接口限流顺序已覆盖：写请求会先按客户端来源计数，触发 429 时短路租户查询；限流未命中时未知店铺仍返回 404；缺失或无效 initData 不进入业务服务；创建订单成功路径再按 `buyer:{telegram_user_id}` 计数；支付创建在订单归属校验通过后才按 `order:{out_trade_no}` 计数，跨买家请求不会污染订单维度限流桶。
- 本地外部商品映射和外部源连接只是身份绑定与凭据加密保存能力，不等同于真实 acg-faka/mcy-shop 适配器接入；即使 provider 列表显示 `order/delivery/context` 可用，也只表示本地 provider 对象实现了对应协议，不表示第三方连接已认证、可下单或可拉取发货；Tenant Admin 外部订单/发货入口和后台自动履约都只是调用已注册 provider 的受控编排，不接入具体适配器时不会产生真实第三方操作；当前不校验外部平台凭据，不内置真实第三方网络调用。外部自动履约已能把本地已支付外部映射订单推进到“创建外部订单、拉取发货、导入本地待发送发货记录”的服务闭环，但真实 provider 接入前仍需确认上游 `out_trade_no` 幂等、错误脱敏和联调策略。

### 9.5 业务插件扩展

平台需要预留正式业务插件架构，用于后续扩展支付通道、外部发卡源/代发上游和可选 Bot/Web/租户工具功能。长期目标接近 WordPress 插件式业务扩展，例如以后接入 TRX 兑换、通知、运营工具或新的后台面板时，应走平台扩展点而不是直接改核心订单和租户逻辑。该目标态不是当前交付范围；当前只做受控 `payment` / `external_source` 适配器插件和最小 manifest 合同。插件不是绕过平台核心业务的独立脚本，目标态必须经过平台注册、能力声明、租户启用、权限校验、凭据加密、审计和离线合同测试。

当前阶段不建设 WordPress 式通用插件运行时。通用运行时会牵动在线安装/升级、插件自带迁移、动态前端面板、动态 Bot handler、任意后台任务、Webhook handler 和权限沙箱，范围超过当前多租户发卡 SaaS 的主线。当前必须冻结的是受控适配器边界：新增支付系统只能通过支付 provider、`payment` manifest、配置 schema 和离线合同进入；新增 acg-faka、mcy-shop 等外部货源只能通过 `external_source` provider、外部源连接、签名 handle、目录同步/下单/发货合同和脱敏 BFF 进入。核心订单、商品、账本和租户隔离流程不得直接依赖具体第三方项目适配器。

当前阶段只落地受控适配器插件和最小 manifest 合同：`app/services/business_plugins.py` 定义 `BusinessPluginManifest`、`BusinessPluginRegistry`、entrypoint allowlist 判定，以及从现有 payment/external_source provider 摘要转换为业务插件 manifest 的静态能力。它不导入执行第三方插件代码，不做动态 Bot router 注入，不做动态任务和 handler 注入，也不做插件市场、远程 JS 或在线安装。

Admin Web 插件能力摘要只读 BFF 已接入：`GET /api/v1/admin-web/business-plugins/capabilities` 只从 httpOnly cookie 当前工作区解析平台或克隆 Bot 上下文，返回静态 manifest 的浏览器安全摘要、当前工作区非密文支付启用状态和外部源连接计数。它不执行插件 entrypoint，不导入远程代码，不读取或解密外部源凭据，不调用 provider 目录/下单/发货方法，不返回 `tenant_id`、连接 ID、凭据字段、raw payload、外部订单 ID、发货内容或 storage key；这不代表插件安装、租户级启停、真实 mcy-shop/acg-faka、真实支付网关或 staging 验证完成。`web/admin` 右侧“插件能力”卡片只做观测，支持独立刷新、加载、错误和空态。

Admin Web 外部源连接管理合同已接入：`GET /api/v1/admin-web/tenant/external-source-connections`、`POST /api/v1/admin-web/tenant/external-source-connections` 和 `POST /api/v1/admin-web/tenant/external-source-connections/disable` 只从 httpOnly cookie 当前克隆 Bot 工作区解析租户，写操作必须通过 Origin 门禁并校验 `settings` 权限。响应只返回 provider 能力摘要、连接展示名、provider/source、状态、时间、`credential_field_count` 和签名 `connection_handle`，不返回内部 `tenant_id`、连接 ID、凭据字段名、明文、密文、API key、secret、token、raw payload、外部订单 ID、发货内容或 storage key；创建连接只做本地 provider 凭据格式校验和加密保存，停用连接只按 `connection_handle` 定位当前租户连接。`web/admin` 已接入连接刷新、创建连接和停用连接；该入口不读取或解密外部源凭据，不调用 provider 目录/下单/发货方法，不触发真实上游联调，也不代表 acg-faka/mcy-shop 真实连接已认证。

Admin Web 外部源目录同步合同已接入：`POST /api/v1/admin-web/tenant/external-sources/catalog/sync` 只从 httpOnly cookie 当前克隆 Bot 工作区和签名 `connection_handle` 解析租户、provider、source 和连接，请求仅允许 `connection_handle`、`cursor`、`limit`、`max_pages`，不接受内部 `tenant_id`、`connection_id`、`provider_name`、`source_key` 或外部商品 ID。服务端校验 active 连接后才加载 runtime credentials，并复用 `ExternalCatalogSyncService.sync_registered_catalog()` 同步一页或少量分页商品目录；响应只返回 provider/source、同步计数、next cursor 和本地商品安全摘要，不返回 `tenant_id`、`connection_id`、`external_id`、`raw_payload`、`credentials`、`token`、`secret`、`storage_key`、`delivery` 或发货内容。`web/admin` 外部源连接行已提供“同步目录”按钮，成功后刷新连接、插件能力和当前工作区商品摘要；该能力不代表真实 acg-faka/mcy-shop 上游联调完成。

插件类型：

- `payment`：新增支付通道，必须声明创建支付、回调验签、可选查单/对账、配置 schema、敏感字段脱敏和生产状态。近期支付主线仍只聚焦 `epusdt_gmpay` 和 `epay_compatible` 的配置入口、离线合同和 staging 验证路径；其他支付插件默认后置，先以 `production_ready=false` / `staging_verified=false` 进入能力列表。
- `external_source`：接入外部发卡源和代发上游，必须实现商品目录、外部下单、查单、取发货、凭据 schema、错误分类和敏感字段脱敏。acg-faka 与 mcy-shop 应优先作为此类插件接入。
- `bot_feature`：扩展 Bot 功能，例如自动客服、通知、文件处理或其他租户功能，必须挂在租户功能开关、RBAC、审计和限流之下；当前阶段只预留 kind，不实现动态 router 注入。
- `admin_web_panel`：给 Web 管理后台增加配置或观测面板；当前阶段只预留 metadata 边界，由核心前端按已知组件渲染，不执行远程 JS。
- `tenant_tool` / `background_job` / `webhook_handler`：目标态业务扩展，例如 TRX 兑换、链上观察、定时同步或外部通知回调；当前阶段只预留 kind，不实现动态任务和 handler 注入。

acg-faka / mcy-shop 接入采用“外部货源插件 + 本地代理商品”模型：租户创建外部源连接并加密保存凭据，插件同步上游商品安全摘要到本地商品映射，租户选择商品上架；买家付款成功后由后台外部履约 worker 使用本地订单号向上游下单、拉取发货并导入本地待发送发货记录。买家侧、Public Store、租户 Bot 和代理商品公开响应不得暴露上游平台、上游店铺、外部商品 ID、外部订单 ID、raw payload、连接 ID 或凭据字段。

自动履约当前运行时只按 provider registry 摘要检查 context 型订单/发货能力和 provider 对象 `auto_fulfillment_idempotent=True`。后续动态插件接入前，还必须补 manifest 门禁、租户级启停门禁、离线重复 `out_trade_no` 回放测试和真实 staging 幂等验证；否则只能做目录同步或人工触发。

详细方案见 `docs/业务插件架构方案.md`。真实 acg-faka/mcy-shop 接入需要先固定目标版本、确认官方 API/插件协议、建立 fixture 合同和脱敏测试；真实上游联调、真实凭据、真实下单和取发货必须由用户明确确认后再执行。

### 9.6 外部项目参考：GAFBot

GAFBot 可以作为“Telegram Bot 文件任务处理体验”的参考，但不作为本平台的架构底座。

可参考的部分：

- Bot 菜单进入具体功能后，再进入等待上传文件状态。
- 使用 document 上传 zip 文件。
- 对同一用户的操作进行队列化，避免并发点击和连续上传导致状态混乱。
- 长任务显示进度、超时、成功、失败状态。
- 处理结果可以打包为 zip 返回给用户。
- zip 解压前检查路径，避免 `../` 路径穿越。
- 使用临时目录处理文件，任务结束后清理。

不能直接照搬的部分：

- 账号登录、验证码读取、2FA、tdata/session 转换、防找回、Passkey 等账号会话操作。
- 使用 JSON 文件保存用户、订单和状态。
- 单 Bot polling 架构。
- `.env` 进入仓库、支付请求关闭证书校验等安全做法。
- 把账号会话凭据类文件作为默认商品能力。

本平台吸收的设计结论：

- 通用文件商品可以支持 zip/rar/7z 上传和交付。
- 文件处理必须租户隔离、异步执行、持久化保存、可审计。
- 文件处理任务使用统一的 `file_processing_jobs` 模型记录状态。
- 对压缩包内部文件做基础安全扫描和敏感路径检测。
- Telegram 账号会话包即使外层是 zip，也不纳入平台内置商品类型。

## 13. 安全设计

### 13.1 租户隔离

所有业务表必须带 `tenant_id` 或明确的供应商/代理商租户字段。

Repository 层必须默认按 `tenant_id` 查询，不允许 Router 直接拼 SQL。

关键隔离点：

- 商品隔离。
- 卡密隔离。
- 用户订单隔离。
- 支付配置隔离。
- Bot Token 隔离。
- 文件和上传资源隔离。
- 审计日志隔离。

### 13.2 敏感数据

必须加密：

- Bot Token。
- 支付密钥。
- 卡密内容。
- 外部系统 API Key。
- 提现地址可不加密但建议脱敏展示。

禁止：

- 日志输出完整 Token。
- 日志输出完整卡密。
- 公开 API 返回卡密。
- 将支付密钥传给 Cloudflare Workers 前端。

### 13.3 风控

需要支持：

- 商品审核。
- 违规商品下架。
- 租户冻结。
- 用户投诉。
- 退款/争议流程。
- 支付回调幂等。
- 订单异常告警。
- 订单创建前频率/金额阈值拦截和审计。
- 订单风控多次命中后的自动处置，默认关闭且可配置；首版只复用平台用户封禁，不自动退款、不自动冻结资金、不做链上动作，解除仍由平台管理员执行。
- Token 泄露重置流程。
- API 限流。
- API IP/CIDR 白名单。
- 黑名单。

### 13.4 文件安全

如果支持上传普通文件商品：

- 文件必须按租户隔离存储。
- 文件名不能直接使用用户上传原名作为存储路径。
- 下载必须走鉴权。
- 平台管理员可审计，但普通租户不能跨租户访问。
- 单个文件最大 5MB。
- 存储路径建议为 `/app/storage/tenants/{tenant_id}/uploads/{yyyy}/{mm}/{random_key}`。
- Docker 部署时将宿主机 `./storage` 挂载到容器 `/app/storage`。
- 数据库保存文件元数据和 `storage_key`，不把文件二进制写入 PostgreSQL。
- 文件下载链接必须有短期有效期，不能是永久公开直链。
- 压缩包解压必须防路径穿越，禁止绝对路径、`../`、符号链接逃逸。
- 压缩包要限制文件数量、解压后总大小、单文件大小和嵌套层级。
- 压缩包处理必须在临时目录执行，任务结束后清理。
- 文件处理任务需要超时控制，避免长时间占用 Bot 更新处理链路。

关于 Telegram 账号会话文件：

- 不建议把 `.session`、`tdata`，或包含这类内容的 zip/rar/7z 压缩包作为平台内置商品类型。
- 这类文件本质上可能包含可直接登录账号的会话凭据，风险高于普通数字商品。
- 外层打成压缩包不会改变风险属性，仍应按账号会话凭据处理。
- 平台风控应至少允许将此类文件类型和压缩包内的敏感路径列入敏感/禁止上传规则，避免平台被用于售卖第三方账号会话或被盗凭据。

## 17. 主要风险和处理

### 17.1 Token 泄露

处理：

- 加密存储。
- 日志脱敏。
- Webhook 不包含 Token。
- 提供 Token 重置流程。

### 17.2 资金争议

处理：

- 平台托管场景必须有账本。
- 提现人工审核。
- 保留订单、回调、账本审计日志。

### 17.3 库存超卖

处理：

- PostgreSQL 行级锁。
- 下单锁库存。
- 超时释放。
- 支付成功后幂等发货。

### 17.4 租户数据串号

处理：

- 所有 Repository 默认带租户过滤。
- 测试覆盖跨租户不可见。
- 管理员跨租户查询必须显式使用平台权限。

### 17.5 第一版范围失控

处理：

- 第一版只做平台内核、自营发卡、订阅、epusdt。
- 供货代理作为第二里程碑；当前已落地 Tenant Admin 代理商侧、Tenant Admin 供应商侧供货 API、独立代理规则最小 HTTP 切片和 Platform Admin 供货商品状态管控最小 HTTP 切片，平台审核和真实分账联调仍后续。
- 外部系统接入只预留接口，不在 MVP 深入实现。

## 18. 已确认业务决策

本节记录已确认的实现口径，后续开发应按这些口径设计。

- 代理商可以基于供应商成本价自行加价销售。
- 供应商可以限制最低售价；如果不设置最低售价，允许代理商低于供应商建议零售价销售。
- 暂时不做多级佣金。
- 供应商和代理商余额通过平台账本记录。
- 提现采用人工审核，不做自动链上转账。
- 提现申请会通过 Bot 通知平台管理员处理。
- 需要风控能力：商品审核、违规商品下架、投诉/退款、订单争议、支付回调幂等、库存锁定超时释放、Token 泄露重置流程。
- 暂时保留 CNY 和其他币种字段；后续若接入 USDT/CNY 汇率 API，可支持按偏好切换显示。
- 当前默认不设置平台手续费，只采用月租收费。
- 平台手续费能力保留；若未来启用，默认 1%，并支持平台管理员修改。
- 供货商品不需要平台审核后才能被代理，但平台保留事后下架和冻结能力。
- 代理商品允许隐藏供应商身份，买家侧和公开 API 不暴露上游供应商。
- Web 管理后台纳入近期独立里程碑，使用 `shadcn/ui`，并必须通过 Bot 工作区选择确定租户上下文。
- 代理商在网页供货市场选择商品并上架前，必须先绑定或选择目标克隆 Bot。
- 近期支付通道只重点推进 `epusdt_gmpay` 和 `epay_compatible`/易支付兼容，其他支付系统对接后置。
- 退款按各方实际获得金额扣回。
- 平台托管订单默认 7 天款项冻结期，冻结期内不能提现；冻结天数由平台管理员配置。
- 租户上传普通文件单个最大 5MB。
- 文件存储由系统设计为 Docker 持久化挂载目录，建议 `./storage:/app/storage`。
- 订阅欠费后数据保留 1 个月。
- 平台需要导出账单和订单报表。
- 第一版需要租户管理员角色，并通过 RBAC 控制权限。

剩余建议在详细设计阶段确认：

- 报表导出的文件格式优先级：CSV、XLSX，或两者都支持。
- 欠费 1 个月后的数据处理是自动软删除，还是进入平台管理员待删除队列。
- 文件类商品第一版已限制为平台白名单扩展名和 MIME 类型，缺失 MIME 或通用二进制 MIME 仅允许随 zip/rar/7z 扩展名进入后续扫描。

## 19. 推荐结论

方案可行，但应按“平台内核优先、交易能力后置、外部生态最后”的顺序推进。

最小可交付版本应聚焦：

- 多租户 Bot 克隆。
- 自营发卡。
- epusdt/USDT 支付。
- 平台订阅收费。
- 安全隔离。

供货代理、账本提现、Cloudflare Workers 网页发卡和外部发卡系统适配都应建立清晰扩展点，但不建议塞进第一版一次性交付。
