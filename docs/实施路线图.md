# 多租户发卡平台实施路线图

## 1. 目标

本文档把 `docs/多租户发卡平台完整方案.md` 拆成可执行里程碑，供后续开发和 AI 接手时按顺序推进。

核心原则：

- 平行新建 `app/` 目标架构，旧版 `bot.py`、`user_flow.py`、`admin_panel.py` 先保留为业务参考。
- 先实现多租户平台内核，再迁移自营发卡，再实现订阅和支付，最后做供货代理、账本和外部接口。
- 每个阶段必须有可验收结果，避免长时间处于不可运行状态。
- 外部货源、支付通道和可选 Bot/Web 功能按业务插件架构扩展，详见 `docs/业务插件架构方案.md`；长期目标接近 WordPress 插件式业务扩展，但当前阶段只落地受控 `payment` / `external_source` 适配器和最小 manifest 合同，真实第三方联调必须独立确认。当前阶段不建设在线安装/升级、动态前端面板、动态 Bot handler、任意后台任务或插件自带迁移；必须先把支付 provider、外部源 provider、manifest registry、租户连接、签名 handle 和脱敏 BFF 边界固定下来，避免后续接 acg-faka、mcy-shop 或新支付通道时改动核心订单、商品、账本和租户隔离流程。
- Admin Web 外部源目录同步合同已接入 `POST /api/v1/admin-web/tenant/external-sources/catalog/sync`：请求仅允许 `connection_handle/cursor/limit/max_pages`，从 cookie 当前工作区和签名 `connection_handle` 解析租户/连接，响应不返回 `tenant_id/connection_id/external_id/raw_payload/credentials/token/secret/storage_key/delivery`；这只是受控适配器 BFF 和离线合同，不代表真实上游联调完成。
- Admin Web 外部源已同步商品列表已接入 `GET /api/v1/admin-web/tenant/external-sources/catalog/products`：请求仅允许 `connection_handle/limit/offset`，从 cookie 当前工作区和签名 `connection_handle` 解析租户/连接，只读取已同步落库的本地商品摘要，不解密凭据、不调用 provider、不返回 `tenant_id/connection_id/external_id/external_source/raw_payload/credentials/storage_key/发货内容`；`web/admin` 外部源连接行提供“查看已同步商品”只读入口。
- Admin Web 平台 Bot Webhook 重置已接入 `POST /api/v1/admin-web/platform/bots/{tenant_public_id}/webhook/reset`：平台管理员会话 + Origin 门禁，请求仅允许 `reason`，只按 `tenant_public_id` 定位 active Bot；服务端调用 Telegram `setWebhook` 成功后才轮换 Webhook secret、清 Redis 缓存、写审计和提交，失败不提交、不轮换、不泄露 Bot Token、Webhook secret 或 raw payload。当前只用 fake Bot 测试，真实 Telegram 操作仍需用户确认后执行。
- Admin Web 平台订阅计划编辑已接入 `PATCH /api/v1/admin-web/platform/subscription/plans/{plan_code}`：平台管理员会话 + Origin 门禁，请求仅允许 `name/monthly_price/currency/trial_days/grace_days/reason`，主 Bot 平台订阅计划面板支持创建、编辑、启用和停用套餐；响应不返回内部套餐、租户、订阅、账单、订单或支付字段，编辑不重写历史 invoice、不创建续费订单、不触发支付建链。
- Admin Web 平台 Bot 状态和供货商品状态写操作已补离线安全合同：`PATCH /api/v1/admin-web/platform/bots/{tenant_public_id}/status` 只更新本地 Bot 状态、审计和本地缓存，不调用真实 Telegram；`PATCH /api/v1/admin-web/platform/supply/supplier-offers/{supplier_offer_id}/status` 只做供货商品软下架/恢复，不删除商品、不触发真实分账。两类接口都使用平台管理员 httpOnly 会话、Origin 门禁、请求白名单和安全响应 DTO。
- Admin Web 剩余平台写接口安全合同已补：用户封禁/解封、租户冻结/恢复、平台提现完成/拒绝、订阅计划创建和订阅计划启停均不声明 Tenant/Platform API Key 或 HMAC 头，request schema 禁止额外字段，response schema 不暴露内部租户、Bot、账本、套餐、订阅、订单、支付、凭据、raw payload 或完整提现地址字段；用户封禁/解封、租户冻结/恢复、平台提现完成/拒绝、订阅计划创建和订阅计划启停已补 route 级离线测试。
- Admin Web 已统一 422 校验错误脱敏：路由层保留错误类型、位置和消息，但剥离 `input/ctx/url`，防止恶意额外字段中的 token、secret、raw payload、提现地址或外部源凭据通过验证错误响应泄露。
- Admin Web 平台提现单笔详情已接入 `GET /api/v1/admin-web/platform/finance/withdrawals/{withdrawal_id}`：平台管理员 cookie 会话只读查询，复用账本服务和安全提现 DTO，不使用 Platform Admin API Key/HMAC，不触发真实打款，前端主 Bot 提现审核卡片可行内查看脱敏详情。
- Admin Web 平台租户订阅调整已接入 `POST /api/v1/admin-web/platform/tenants/{tenant_public_id}/subscription/grant-days` 和 `PATCH /api/v1/admin-web/platform/tenants/{tenant_public_id}/subscription/period-end`：平台管理员 cookie 会话 + Origin 门禁，只用公开 `tenant_public_id` 定位租户，复用 `SubscriptionService.grant_days/set_period_end`，不接受内部租户/订阅/套餐/账单/订单 ID，不创建续费订单、不支付建链、不外联；前端主 Bot 订阅关注项提供“赠送 30 天”和“设置到期”入口。
- Admin Web 平台租户/Bot 列表服务端分页筛选已接入 `GET /api/v1/admin-web/platform/dashboard`：平台管理员 cookie 会话只读查询，支持 `tenant_limit/tenant_offset/tenant_query/tenant_status/bot_status/subscription_status`，搜索范围限定店铺公开名、租户公开 ID、Bot 用户名和 owner 用户名；响应仍只返回安全摘要，不返回内部租户/Bot/订阅 ID、Bot Token、Webhook secret 或 raw payload；前端主 Bot “租户与 Bot”面板已接入搜索、状态筛选和上一页/下一页分页。

## 2. 开工前固定决策

- 技术栈：Python 3.11+、aiogram 3.x、FastAPI、PostgreSQL、Redis、SQLAlchemy 2.x async、Alembic、Pydantic Settings。
- 运行模式：生产使用 Webhook，多 Token 单进程分发。
- 租户单位：一个克隆 Telegram Bot 对应一个租户。
- Token 来源：第一版使用 BotFather 手动 Token。
- Token 存储：加密保存，日志脱敏，Webhook URL 不包含原始 Token。
- 开发策略：新建 `app/` 平行开发，不直接在旧大文件里硬改。
- 文件商品：支持普通 zip/rar/7z 文件商品，不内置支持 Telegram 会话凭据包。
- 资金策略：默认月租收费，不启用平台抽成；抽成能力保留，默认 1%。
- 结算策略：平台托管订单默认 7 天冻结期。

## 4. 阶段 1：多租户平台内核

### 目标

母 Bot 能绑定租户 Bot，Webhook 能按 secret 找到租户并分发 Update。

### 任务

- 实现基础表：
  - `platform_users`
  - `tenants`
  - `tenant_bots`
  - `tenant_settings`
  - `tenant_members`
  - `audit_logs`
- 实现 Token 加密服务：
  - `encrypt_token`
  - `decrypt_token`
  - `token_hash`
  - 日志脱敏工具
- 实现母 Bot Router：
  - `/start`
  - 创建/绑定机器人
  - 查看我的机器人
  - 重置 Webhook
  - 停用 Bot
- 实现 Webhook：

```text
POST /telegram/webhook/{webhook_secret}
```

- 实现 `TenantContextMiddleware`：
  - 注入 `tenant_id`
  - 注入 `tenant_bot_id`
  - 注入 `owner_user_id`
  - 注入租户设置和功能开关
- 实现 Redis 缓存：
  - `webhook_secret -> tenant_bot`
  - 租户设置
- 当前 Webhook 会按 `webhook_secret` 解析并缓存 `tenant_context`；已有运行时测试覆盖 master secret 分发、租户 secret 命中 Redis 缓存后仍回源确认 Bot 当前 active 再分发 tenant context、停用或轮换后残留 Redis 缓存会被删除且不再分发、Redis miss 经 DB 回源并以 300 秒 TTL 写回缓存、旧格式 Redis 缓存失效后回源 DB 并重新缓存。HTTP 层负向 smoke 已覆盖 dispatcher 未初始化返回 503、MASTER_BOT_TOKEN 缺失返回 503、未知租户 secret 返回 404、dispatcher 分发异常返回 500 且仍关闭 Bot session；成功和失败响应均断言不回显原始 Bot Token、密文 Token 或 Webhook secret。`TenantContextMiddleware` 会在租户 Update 中注入 `tenant_settings` 和 `tenant_feature_flags`，店铺首页、客服和设置展示会优先使用已注入的租户设置，减少重复查询。
- 已补平台运行时离线 E2E smoke：在同一个 fake FastAPI app 中验证 `/health`、`/ready`、租户 Webhook secret Redis 命中、Token 解密创建 Bot、fake dispatcher 通过 `TenantContextMiddleware` 注入租户设置和 feature flags，并断言响应不泄露原始 Bot Token 或密文；该 smoke 不触发真实 startup、不连接真实 Redis/PostgreSQL、不调用真实 Telegram。
- 已补 `create_app()` 组合层离线 smoke：patch `get_settings()`、`configure_db()`、`redis.from_url()`、`create_dispatcher()`、`BackgroundWorkerManager` 和 `close_db()`，验证核心路由子集已挂载，startup 会注入 dispatcher/Redis/worker 并调用 worker `start()`，`/ready` 会使用 lifespan 注入的 Redis 并调用 worker readiness，shutdown 会 stop worker、关闭 Redis 并调用 `close_db()`；OpenAPI 中 `/health` 和 `/ready` 保持公开，不继承 Tenant Admin 鉴权或签名头。
- 母 Bot 已支持 `/mybots` 展示租户 Bot ID 和状态，owner 可用 `/reset_webhook BotID` 轮换租户 Webhook secret 并重新设置 Telegram Webhook，可用 `/deactivate_bot BotID` 停用租户 Bot 并删除其 Telegram Webhook；两类操作均按 owner 隔离、写审计且清理旧 `tenant_webhook:*` 缓存。
- 租户 Bot `/start` 已补离线 smoke：缺失 tenant context 时直接提示店铺暂不可用且不查仓储；空店铺会使用 tenant context 店铺名和默认欢迎语，普通买家只看到商品列表、订单和客服入口，owner/admin 才看到商家管理入口；`tenant:home` callback 缺失 tenant context 时只确认 callback 不渲染页面。
- 租户 Bot 管理入口已补 handler 级离线 smoke：非 owner/admin 直接访问 `/manage` 或 `tenant:manage` callback 会提示无权限，不渲染商家管理菜单；群聊中的管理入口仍先提示私聊使用并跳过权限查询。
- 租户 Bot 管理命令和管理 callback 已统一要求私聊上下文；群聊/超群中会先提示“管理功能请在私聊中使用”，不会继续查询权限或执行管理动作，买家侧公开浏览与下单入口不受影响。
- 实现基础 RBAC：
  - `owner`
  - `product_admin`
  - `order_admin`
  - `finance_admin`
  - `support_admin`

### 验收

- 母 Bot 可绑定一个测试 Bot Token。
- 平台能调用 Telegram `getMe` 验证 Token。
- 数据库不保存明文 Token。
- Webhook URL 不包含 Token。
- 克隆 Bot 收到 `/start` 时能识别正确租户。
- 非 owner 无法进入租户管理。
- 停用租户后，该 Bot 的普通买家请求被拒绝或提示不可用。

### 验收

- 租户 A 看不到租户 B 的商品和库存。
- 卡密支付成功只发一次。
- 支付超时释放锁定库存。
- 文件商品支付成功后可下载，链接过期后不可下载。
- 上传文件不因容器重启丢失。
- 上传入口已限制为 zip/rar/7z 压缩包：非白名单扩展名或明确 MIME 冲突会在下载 Telegram 文件前拒绝，缺失 MIME 或通用二进制 MIME 仅在扩展名为 zip/rar/7z 时继续进入后续文件头和压缩包扫描。
- 文件扫描已通过线程池执行并加入超时控制，超时会阻断绑定；扫描规则已阻断 zip 路径穿越、Windows 绝对路径、过多条目、超大单文件、超大解压总量、过深嵌套、Telegram `.session`/`.session-journal`、`tdata`/`key_data`/`auth_key` 会话凭据，以及常见被盗凭据线索如 `.env`/`.env.*`、`.netrc`、`cookies.txt`、SSH 私钥、AWS/GCloud 凭据路径；RAR/7Z 已做无依赖壳层校验：文件头、扫描层大小上限、过短魔数文件和外层敏感文件名会被拦截，合法壳层仍标记为 medium，条目级深度扫描后续单独评估。
- Telegram 文件下载失败时会删除本地 `.tmp` 临时文件，不在租户文件目录残留半文件。
- 租户 Bot 已支持 `/export_inventory 商品ID [数量]` 在私聊导出 `card_pool/card_fixed` 商品的可用卡密明文 TXT；命令要求商品库存权限，单次最多 5000 条，导出动作只写数量和商品 ID 审计，不记录卡密内容。
- 商品分类和排序已开放租户 Bot 命令 `/set_product_category 商品ID | 分类`、`/set_product_sort 商品ID | 排序值`；商品列表继续按 `sort_order ASC, created_at DESC` 返回，分类字段同时支持 Tenant Admin 创建/同步和外部目录同步元数据映射。
- Admin Web 自营库存导入已接入：`POST /api/v1/admin-web/tenant/products/{product_id}/inventory/import` 只从 cookie 当前工作区解析克隆 Bot 租户，写操作必须通过 Origin 门禁，请求仅允许 `items`，只支持 `card_pool` / `card_fixed` 自营文本库存；后端会去空行、去输入内重复、加密并哈希后写入库存，响应只返回导入数量、重复数量和可用库存数，不返回库存明文、密文、hash、storage key 或库存项 ID。

## 12. 阶段 8P：业务插件架构

### 目标

在现有 `external_sources` 和 `payments` 抽象上补业务插件 manifest、能力声明和静态 registry 合同，使后续支付通道、acg-faka/mcy-shop 等外部货源可以先按受控适配器扩展，并为可选 Bot/Web/租户工具类插件保留目标态边界。

### 任务

- 设计插件 manifest：`plugin_id`、类型、版本、入口、合同版本、能力、`production_ready`、`staging_verified`。
- 当前阶段落地受控适配器插件：`payment`、`external_source`；`bot_feature`、`admin_web_panel`、`background_job`、`webhook_handler`、`tenant_tool` 仅作为目标态 kind 和安全边界预留。
- 第一阶段只做本地静态 registry、entrypoint allowlist 判定和现有 provider 能力摘要转换，不导入执行第三方插件代码，不做在线插件市场、远程代码执行或自动更新。
- 为 acg-faka、mcy-shop 建立外部货源插件候选方案：先做 fixture 合同、字段映射、错误脱敏、敏感 raw payload 拒绝和重复 `out_trade_no` 回放测试，再进入真实 staging。
- Admin Web 后续增加插件能力摘要、连接配置、目录同步和商品上架入口，但不得展示上游 API Key 明文、外部订单 ID、发货内容、raw payload 或 storage key。

### 验收

- 当前阶段插件 registry 能列出插件 ID、类型、版本、合同版本、能力、`production_ready` 和 `staging_verified`。
- 目标阶段禁用或未安装插件不会出现在租户外部源连接页，也不会被后台 worker 调用。
- 外部货源插件不向买家侧、Public Store、租户 Bot 或代理商品公开响应暴露上游平台、上游店铺、外部商品 ID、外部订单 ID、raw payload、连接 ID 或凭据字段。
- 自动履约插件必须同时通过离线幂等证明和真实 staging 幂等验证，才能声明 `auto_fulfillment_idempotent=true`。

## 14. 每阶段通用验收

每阶段结束必须检查：

- 数据是否按 `tenant_id` 隔离。
- 非 owner 或无权限管理员不能访问管理功能。
- 日志不输出 Token、支付密钥、卡密、敏感文件路径。
- 数据库迁移可重复执行。
- 数据库迁移至少通过 `python scripts/verify_migrations.py` 的离线验证；真实 `alembic upgrade head` 需在确认目标数据库后执行。
- Docker Compose 能启动。
- 最少有 smoke 测试覆盖主流程。

## 15. 不做清单

- 多级佣金。
