# 阶段 0-1 启动说明

本文说明新建的 `app/` 多租户平台内核如何启动。旧版 `bot.py`、`user_flow.py`、`admin_panel.py` 暂时保留为业务迁移参考。仅修改代码、执行编译检查和导入检查时不需要创建 `.env`；只有启动真实服务、绑定 Telegram Bot 或接入支付时才需要填写实际配置。

## 1. 准备环境变量

复制环境变量模板：

```bash
cp .env.example .env
```

生成 `TOKEN_ENCRYPTION_KEY`：

```bash
python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

启动真实服务时必须填写：

- `PUBLIC_BASE_URL`：公网 HTTPS 地址。
- `MASTER_BOT_TOKEN`：母 Bot Token。
- `MASTER_WEBHOOK_SECRET`：母 Bot Webhook secret，默认 `master`。
- `TOKEN_ENCRYPTION_KEY`：稳定保存，丢失后无法解密已绑定租户 Token。
- `PLATFORM_ADMIN_IDS`：平台管理员 Telegram 用户 ID，多个用英文逗号分隔。

Telegram WebApp 可选安全配置：

- `TELEGRAM_WEBAPP_REQUIRE_INIT_DATA=true`：生产 Telegram WebApp 下单/支付入口建议开启。
- `TELEGRAM_WEBAPP_INIT_DATA_MAX_AGE_SECONDS=86400`：限制 initData 的 `auth_date` 最大有效期。

## 2. 启动基础服务

```bash
docker compose up -d postgres redis
```

执行数据库迁移：

```bash
docker compose run --rm app alembic upgrade head
```

启动应用：

```bash
docker compose up -d app
```

检查状态：

```bash
curl http://127.0.0.1:58001/health
curl http://127.0.0.1:58001/ready
```

`/health` 只表示应用进程存活；`/ready` 会检查 PostgreSQL、Redis 和启用状态下的后台 worker readiness。`WORKERS_ENABLED=true` 时，如果 `BackgroundWorkerManager` 缺失、未实现 readiness 接口或已有后台 task 退出，会返回 `503 worker_unavailable`；`WORKERS_ENABLED=false` 时跳过 worker readiness。Compose 的 app healthcheck 使用 `/ready`，依赖或后台 worker 未就绪时不应把容器标记为 healthy。

## 2.1 开发验证

仅做代码修改时不需要填写 `.env`。可先执行轻量检查：

```bash
".venv/bin/python" -m compileall -q "app" "alembic" "tests" "scripts"
".venv/bin/python" -m unittest discover -s "tests" -p "test_*.py"
".venv/bin/python" -m unittest "tests.test_health_readiness" "tests.test_app_runtime_smoke" "tests.test_worker_scheduler_readiness" "tests.test_staging_readiness"
".venv/bin/python" "scripts/staging_readiness.py"
".venv/bin/python" "scripts/verify_migrations.py"
docker compose --env-file ".env.example" config
```

`scripts/verify_migrations.py` 只检查模型 metadata、迁移 revision 链、Alembic head，并生成 PostgreSQL 方言离线 SQL 到 `/tmp/fakabot_alembic_head.sql`；它不会连接数据库，也不会执行 `alembic upgrade head`。真实数据库迁移属于结构变更，执行前必须确认目标环境和备份策略。

如果本机未安装 `requirements.txt` 里的依赖，请在 Docker 镜像或项目虚拟环境中运行 `unittest`；`compileall`、迁移离线验证和 `docker compose config` 可用于先发现语法、迁移链与 Compose 配置问题。

## 3. 设置母 Bot Webhook

母 Bot Webhook 地址格式：

```text
{PUBLIC_BASE_URL}/telegram/webhook/{MASTER_WEBHOOK_SECRET}
```

租户 Bot 绑定成功后，系统会为租户 Bot 自动设置：

```text
{PUBLIC_BASE_URL}/telegram/webhook/{tenant_webhook_secret}
```

  - `LedgerService.audit_account_balance()` 提供只读账户余额审计，可按 `ledger_entries` 重算 pending/available/frozen 并和 `ledger_accounts` 快照对比
  - 已补提现外部打款流水和凭证链接字段；提现完成审计和租户通知会展示打款凭证摘要
  - 已补账本/提现/退款表的基础数据库 `CHECK` 约束：金额必须为正、冻结和待结算余额不得为负、提现/退款状态和账本方向限制在当前服务层状态集合内

  - 租户 owner 使用 `/reseller_products [数量]` 查看已上架代理商品
  - 需要审批的供货商品必须通过审批后才能上架和继续对买家售卖
  - 买家侧 `/products` 同时展示自营商品和隐藏供应商的代理商品
  - 买家使用 `/buy_reseller_product 代理商品ID` 或 inline 按钮购买代理商品

  - 卡密类代理订单下单时锁定供应商库存，支付成功后扣供应商库存并由代理商 Bot 发货

- 订单超时服务：
  - 扫描过期 `pending` 订单
  - 标记订单为 `expired`
  - 释放订单锁定库存
- 后台调度器：
  - 应用启动后周期执行订单过期扫描
  - 周期释放过期库存锁
  - 周期查询待支付 epusdt 订单并补偿支付状态，默认间隔由 `PAYMENT_RECONCILE_INTERVAL_SECONDS` 配置
  - 周期处理已支付外部映射文本商品订单，默认间隔由 `EXTERNAL_FULFILLMENT_INTERVAL_SECONDS` 配置；该任务调用本地已注册 provider 对象创建外部订单、拉取发货并导入本地待发送发货记录，支付回调事务内不直接调用外部源
  - 周期领取 `pending` 发货记录并通过租户 Bot 自动发送，默认间隔由 `DELIVERY_DISPATCH_INTERVAL_SECONDS` 配置；发送失败会标记 `failed`，不主动循环重试 failed 记录；每轮扫描前会按 `DELIVERY_SENDING_TIMEOUT_SECONDS` 把超时 `sending` 记录标记为 `failed`，供商家确认后手动重试
  - 周期释放到期账本结算，默认间隔由 `LEDGER_SETTLEMENT_INTERVAL_SECONDS` 配置
  - 周期处理报表导出任务，默认间隔由 `EXPORT_JOB_INTERVAL_SECONDS` 配置
  - 默认启用，可通过 `WORKERS_ENABLED=false` 关闭；启用时 `/ready` 会要求后台任务已启动且 task 未退出
- Public Store API 基础闭环：

  - `GET /api/v1/store/{tenant_public_id}/products` 返回自营商品和隐藏供应商的代理商品
  - `GET /api/v1/store/{tenant_public_id}/products/{product_id}` 查询公开商品详情
  - `POST /api/v1/store/{tenant_public_id}/orders` 创建自营或代理订单

  - 订单风控已支持按买家维度检查近期下单频率和同币种订单金额阈值，作为平台用户封禁之外的创建前拦截能力；命中时写入 `platform_risk.order_creation_blocked` 审计日志，Public Store 和租户 Bot 只提交该审计记录，不提交普通下单失败路径中的半成品订单状态
  - 订单风控自动处置默认关闭；开启 `ORDER_RISK_AUTO_BAN_ENABLED=true` 后，会按 `ORDER_RISK_AUTO_BAN_WINDOW_SECONDS` 和 `ORDER_RISK_AUTO_BAN_BLOCKED_COUNT_THRESHOLD` 统计同一买家的风控拦截命中次数，达到阈值时复用平台用户封禁并写入 `platform_risk.user_auto_banned` 审计；平台管理员不会被自动封禁，解除仍走 `/unban_user`
  - 平台下架的供货/代理商品不能由商家命令重新开放或重新上架
  - 冻结租户会阻断租户 Webhook、公开店铺 API、Tenant Admin API、供货市场展示和代理订单创建
  - 欠费或冻结租户的买家侧 Bot 商品列表会提示不可下单并隐藏购买按钮；商品订单支付链接创建阶段也会再次校验租户状态
  - 订阅续费不会自动恢复人工冻结状态，必须由平台管理员执行恢复
  - `disputes` 保存订单争议记录，争议处理与账本退款解耦
  - 平台管理员使用母 Bot `/open_dispute 订单号 | 原因` 创建或重开争议
  - 平台管理员使用母 Bot `/disputes [租户ID|状态|all] | [状态|数量] | [数量]` 查询争议
  - 平台管理员使用母 Bot `/review_dispute 争议ID | 备注` 将争议标记为处理中
  - 平台管理员使用母 Bot `/close_dispute 争议ID | resolved/rejected/closed | 处理结论` 关闭争议
  - `after_sale_cases` 保存订单售后工单，可关联平台托管账本退款
  - 平台管理员使用母 Bot `/open_after_sale 订单号 | refund/complaint/reseller_after_sale | 金额或 - | 原因` 创建售后工单
  - 平台管理员使用母 Bot `/after_sales [租户ID|状态|all] | [状态|数量] | [数量]` 查询售后工单
  - 平台管理员使用母 Bot `/review_after_sale 售后ID | 备注` 将售后工单标记为处理中
  - 平台管理员使用母 Bot `/refund_after_sale 售后ID | 金额 | 备注` 执行售后关联退款
  - 平台管理员使用母 Bot `/close_after_sale 售后ID | resolved/rejected/closed | 处理结论` 关闭售后工单
  - 风控动作、争议和售后状态流转都会写入审计日志；不会自动冻结资金或自动链上退款

- 多套餐、优惠券和订阅账单报表。
- 资金冻结和保证金流程。
- 自动链上打款。
